## LoRA 训练曲线诊断

微调训练过程会输出四类关键曲线，应像"体检报告"一样综合看，而非单独判断。

**Training Loss（训练损失）** 是最核心的观察点。健康表现是曲线持续下降，最终贴近 0。若曲线在较高位置（如 0.5）趋于平稳，说明模型还没学透——对于高度结构化的提取任务，理想应是接近 0；可能原因是数据量过少或 epoch 不够，此时可以增加训练步数或数据量再尝试。

**Gradient Norm（梯度范数）** 反映训练稳定性。前几步出现峰值是正常现象（初始学习率较高，在最优解附近震荡），关键是后续有没有发生梯度爆炸——若曲线持续缓降并趋于稳定，说明训练参数温和可持续。这个指标本身信息量有限，主要作辅助参考。

**Learning Rate（学习率调度）** 若是线性衰减至 0，则步数由数据量决定。若 step 数只有 18、epoch 仅 3 次，说明数据量极少（推测 20~50 条），学习率在模型还没学透时就强制归零了——应适当增加 epoch（从 3 调到 4~5）。

**四种典型模式对照**：

| 现象 | 诊断 | 对策 |
|------|------|------|
| 训练 loss 和验证 loss 均上升/不降 | 训练失败（学习率过高，在最优解附近横跳） | 降低学习率 |
| 两者均下降但未贴近底部 | 欠拟合 | 增加数据量 或 增加 epoch |
| 训练 loss 下降，验证 loss 上升 | 过拟合 | 减少 epoch |
| 两者均平稳下降趋近底部 | 训练成功 | — |

**三种模型表现不佳的类型与对策**：格式错误 → 增加数据量；事实性错误 → 放弃微调，改用 RAG；复读/乱码 → 该任务不适合微调。

**Qwen3 深度思考注意事项**：若使用带推理能力的 Qwen3 版本进行微调，训练数据中若缺少 `sync` 标签或数据质量不高，微调过程可能会洗掉模型的深度推理能力，导致输出重复或退化。

## 微调后模型评估方法

微调完成后需同时做**定量评估**和**定性评估**。

**定量评估**的基础是构建一套**锤类测试集**（held-out test set）：100~500 条人工标注的高质量样本，严格不包含训练数据中的任何条目。评测指标使用 **ROUGE**（衡量召回率/覆盖度）和 **BLEU**（衡量精确度/n-gram 重合），两者均反映模型输出与标准答案之间的距离。

当需要批量评测多个微调版本时，人工逐一比较代价过高，此时可使用 **LLM-as-Judge**：用能力更强的模型（如 Qwen-Max）对每条输出按 1~5 分打分，评分维度通常包括准确性、合规性、专业程度。分数是相对上一个版本的自比较，而非绝对标准。

医疗行业有专用评测基准 **MedBench**（或 MedicineBench），法律行业有 **LawBench**，这些领域特定基准内置了经专家标注的测试集和评分模型，可直接复用；其他垂直行业可参考此思路自建。

**定性评估**包含两类：**专家盲测**（邀请领域专家人工检查逻辑错误、事实幻觉、专业名词准确性）和**红队测试（Adversarial Testing）**（故意输入陷阱问题，测试模型的拒绝机制和防御能力，如要求执行危险命令、通过情感操纵绕过安全限制）。

## Agent 安全：Claude Code vs OpenCloud 企业部署

在公司内部推广 AI 编码/执行类 Agent 时，需区分"类 OpenCloud 工具"（如 Claude Code）和 OpenCloud 本身。

**Claude Code 类工具**在企业内部使用需关注两个核心风险：

1. **沙盒安全**：Agent 执行命令需要宽松权限才能足够智能，但权限越高风险越大。解决方案是在受控的沙盒环境中运行，给予最高权限的同时隔离其对宿主系统的影响。

2. **提示词注入（Prompt Injection）**：恶意上下文可能被写入 CLAUDE.md 或记忆文件，导致 Agent 行为偏离预期，功能弱化或上下文混乱。防御方式是定期检查记忆文件，并对 Agent 可读取的外部输入做严格过滤。

**OpenCloud 本身**目前不建议在企业内大规模部署，原因包括：无用户验证机制（所有人共用同一实例）、Plugins/Skills/MCP 均可被 Agent 学习并带来额外安全隐患、代码质量差（完全 AI 生成，难以人工审核）、**token 消耗极快**（3~5 句对话可能烧掉 $1，月套餐需 $150）。建议仅个人使用。

火山引擎有针对 OpenCloud 用户验证问题的插件方案，可作为企业场景下的补丁参考。

## AI 中台建设：应对快速迭代的架构策略

AI 应用层迭代极快（上半年 RAG、下半年 Agentic RAG；LangChain 0.x → 1.0；Dify → LangGraph → AutoGen/CrewAI），但底层能力相对稳定。应对策略是**人为划定稳定层与易变层**。

**稳定层（不变）**：
- MCP 协议与工具接口
- Skills（技能封装）
- RAG 知识库与向量检索
- 用户 Memory 持久化（格式可变，但 memory 本身不变）
- 传统后端 API（支付、数据存储等业务接口）

**易变层（随时可替换）**：
- 业务 SOP 的执行方式（Dify → LangGraph → 未来新框架）
- Agent 与用户的交互形式（react loop、human-in-the-loop 模式）
- 前端呈现层

中台设计的核心原则：将稳定层抽象为可复用基础设施（MCP、Skills、RAG、Memory），暴露统一接口；易变层的 SOP 逻辑和 human-in-the-loop 机制可以随时替换而不影响底层。这样即使上层工具更迭（OpenCloud → Claude Code → 下一代框架），底层积累的工具、知识库、记忆状态都不会浪费。

**GPU 算力平台参考（微调训练用）**：AutoDL（约 2~3 元/小时，有 Ascend L20、RTX 5090、6000 系列）；阿里云、腾讯云、华为云各有公有云 GPU 方案，按实际业务场景选型。
