## 多模态 AI 工业安全应用：小模型 + 大模型协同

工厂摄像头安全监控是一个典型的小模型 + 大模型两阶段架构案例。第一阶段用 YOLO 等目标检测模型做初步识别（如检测火焰、烟雾、电焊等），但小模型误报率高——摄像头难以区分火焰与汽车尾灯，吸烟与电焊。第二阶段将疑似报警的**视频片段**（而非单帧图片）送入大模型做二次审核：火焰在视频中会有跳动特征，尾灯则会随车辆移动形成线条轨迹，大模型结合时序信息可以准确区分。

对于吸烟 vs 电焊的判断，可进一步引入多层规则：先识别人脸区域，再判断该区域附近是否有手部动作，再结合是否佩戴焊接面具（有亮点 + 有面具 = 电焊，有亮点 + 无面具 = 可疑吸烟）。这类问题最终往往不是技术难题，而是成本问题——专用 AI 硬件盒子（含国产 GPU + 量化模型）价格高达十几万，导致很多项目搁置。

## AI 嵌入现有系统的路径

实际落地中，AI 项目几乎都不是从零重建，而是**嵌入到现有系统的某个节点**来试点。从商业角度，公司通常要求先把 AI 嵌入一个具体功能验证效果，而非重构整套系统。

一个典型切入点是将界面交互方式从"按钮点击"改为"语音/文字自然语言输入"。以订票系统为例，用户原来点击出发地、目的地、日期等字段逐一填写，改造后可以直接说"帮我订明天北京到上海的机票"。前端体验大幅提升，而后端驱动的仍然是原有订票逻辑，改动最小。这是 AI 嵌入现有系统性价比最高的切入方式。

## 传统编程与大模型的软硬对照：从订票系统讲起

大模型引入的实质是将传统程序中的"硬性逻辑"替换为"软性逻辑"：

| 传统编程 | 大模型对应 | 说明 |
|---------|-----------|------|
| `if/else` 条件判断 | **意图识别** | 软性分类，判断用户意图而非数值比较 |
| `while` 循环 | **Agent Loop** | 模型自我反思"任务是否完成"，未完成则继续执行 |
| 函数调用 | **Tool Calling → MCP → Skills** | 模型能力触顶时调用外部工具；演化路径：function calling → MCP → prompt+MCP = skills |
| 函数必选参数 | **槽位（Slot）** | 必须收集齐的关键信息（出发地、目的地、日期、乘客）才能驱动后续流程 |
| 人工确认流程 | **Human-in-the-Loop** | 关键步骤暂停等待人工介入（如扫码支付），程序挂起直到用户操作完成 |

这套软性逻辑与传统程序逻辑一一对应，并非取代后者，而是**在程序边界处加了一层自然语言的理解与调度层**，让原有确定性逻辑依然在背后运行。

工程化演进方向是：随着对 agent 的信任度提高，可以逐步将更多中间节点也交给大模型判断——从"只在开头和结尾用大模型"到"整个工作流都由 agent 节点串联"，形成真正的 multi-agent 架构。

## 槽位填充与 Human-in-the-Loop 的实现

**槽位填充（Slot Filling）**的实现机制：提示词驱动模型检查当前已收集的参数是否完整；若缺少必填参数，模型主动向用户追问；用户回复后，上下文历史（已有的 3 个参数）与新提供的第 4 个参数一并送入模型，直到槽位全部填满，才触发后续的确定性程序执行。这是纯上下文积累驱动的，无需显式状态机。

**Human-in-the-Loop** 在 LangGraph 中有专门实现——工作流在需要人工确认的节点暂停（如支付步骤），程序挂起等待；用户完成操作（扫码支付成功）后，工作流恢复并根据结果路由到下一个 agent 节点或错误处理分支。这与传统系统中的回调函数（callback polling）语义相同，但以声明式图结构表达，更易维护。

## Agent 评估框架：业务层 + 技术层

评估一个 agent 的质量分两个维度：

**业务能力评估**：在模拟生产环境（非正式环境，避免真实用户流量打垮 agent）中，用真实的用户查询跑完整工作流，观察：
- Agent 的意图识别路由是否与传统系统一致
- 工具调用成功率（agent 是概率型系统，存在失败概率）
- 输出格式、情感、专业度是否满足业务要求
- 错误率是否在业务可接受阈值内

特别要覆盖输入多样性：标准用语、方言、倒装句、缩略表达、不同语言——这些都是真实用户会输入的内容。

**技术能力评估**：使用现有的 agent 压测和能力评测工具（如腾讯云社区的 LaBenGas 等），测试：
- 标准问答准确率（与基准答案对比）
- 边界条件：上下文被打满时的行为（参考 OpenCloud 被 Meta AI 安全组件触发上下文溢出导致删除邮件的案例）
- 并发压力下的稳定性

业务层测试优先于技术层测试——先确认 agent 能满足真实业务需求，再做技术性能压测。

> [!example]- Answer Guide
> 
> #### Q1 — 两阶段架构与视频片段
> 
> 小模型（如 YOLO）误报率高，无法区分火焰与汽车尾灯等相似场景；将**视频片段**送入大模型后，模型可利用时序信息——火焰有跳动特征、尾灯则形成移动线条轨迹——从而准确区分两者。
> 
> #### Q2 — while 循环与函数调用演化
> 
> `while` 循环对应 **Agent Loop**：模型自我反思"任务是否完成"，未完成则继续执行；"函数调用"的演化路径为 **function calling → MCP → prompt + MCP = skills**，每一步都是在能力触顶时引入外部工具扩展。
> 
> #### Q3 — 槽位填充与 Human-in-the-Loop
> 
> 槽位填充通过**提示词**驱动模型检查已收集参数是否完整，缺少必填槽位时主动追问用户，整个过程靠上下文历史积累驱动，无需显式状态机。Human-in-the-Loop 在 LangGraph 中以**声明式图节点**表达，工作流到达人工确认节点时挂起，用户操作完成后恢复并路由到下一节点，与传统回调语义相同但更易维护。
