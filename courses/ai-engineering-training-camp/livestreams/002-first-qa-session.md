## LangSmith vs LangFuse：企业交付中的可观测性选型

两者都调用相同的 LangChain/LangGraph 接口，差异在于定位：**LangSmith** 是 LangChain 官方维护的在线服务，包含追踪+自动化测评两套逻辑；**LangFuse** 是完全开源的监控工具，仅提供 tracing 和告警，不具备自动化评估能力。

在企业交付场景中，若客户要求量化质量报告才允许上线，LangFuse 无法直接满足；若只需要追踪和调试，LangFuse 完全可胜任，且本地部署成本可控。推荐的实践分层如下：**开发阶段在本机运行 LangSmith**，充分跑通自动化测试、性能评估和调试；**上线后在客户环境改用 LangFuse**，仅保留监控和告警。若生产中出现问题，将相关数据带回本地，继续用 LangSmith 分析——这样既控制了商业授权成本，又保留了完整的评估能力。

## 国产向量数据库替代方案

在国产化要求（中间件/数据库/操作系统/CPU 全部国产化）的场景下，向量数据库有成熟替代品。常见选择：

- **TiDB**（PKF）：内置 TiKV 向量扩展，兼容 MySQL，互联网案例居多
- **OpenMLDB**：同样支持向量化扩展
- **XinHuan（星环）**：政务和金融案例较多

选型时有两个关键注意事项。第一，**性能损耗**：从 Oracle/PostgreSQL 迁移到国产数据库时，性能下降约 30%，且短期难以弥合，因此要做好性能监控，避免向量数据库成为系统瓶颈。第二，**GPU 优先**：向量计算本质上是矩阵运算，GPU 处理远优于 CPU；若有 GPU 版本，尽量在 GPU 上运行，不要跑在 CPU 上。

## 三种场景的模型选型：意图识别、RAG、Agent

**意图识别**的目标是将用户自然语言映射为固定分类（如"授权"/"授后"），核心评估指标按优先级排序为：速度 → 准确性 → 可解释性 → 成本。速度和准确性必须同时达到，否则整个业务流会显得卡顿，且无法用 streaming 掩盖等待感。效果不好时只有两条路：换参数更大的模型，或做微调。推荐模型：BERT-Chinese 系列（对中文意图微调效果稳定）、Qwen 系列；若意图需要从图片中的文字识别，可用 DomoAI 或 Qwen-VL。

**RAG 检索**关注两个环节。检索阶段看 Embedding 模型：语义质量要与存入向量库时一致，且要支持长文档（企业合同、报告等）；推荐 **BGE-M3**（效果优于 OpenAI text-embedding-ada）和 **Jina**（法国公司，中文支持好）。重排阶段看 Reranker：需要支持 Rerank 模式和混合检索（关键词 + 向量），推荐 **BGE Reranker**。

**Agent** 最关键的能力是**工具调用的可靠性**——能否准确识别参数、映射到函数 schema、保持数据类型一致。记忆（短期/长期）可用外部数据库实现，并不依赖模型本身；反思能力次之，主要用于任务分解和工具调用失败后的重试兜底。私有部署推荐 70B 量级（如 Qwen3-72B、DeepSeek-70B），参数过小会导致上下文理解问题，参数过大则消耗 GPU 资源过多、响应速度慢。

## RAG vs 文件上下文（Claude Code 模式）

RAG 与直接把文件塞入上下文（如 Claude Code 文件拖入方式）在架构上有本质区别。

RAG 的核心优势在于**结构清晰、模块解耦**：索引构建（离线）与检索生成（在线）完全分离；可以在文档中附加 metadata（如过期时间），在检索时按 metadata 过滤——例如日期过期的文档片段直接排除，不进入生成阶段；检索结果有完整的来源溯源链路，可解释性强；可扩展性高，可接入 Circle 工具、多策略混合检索等。主要缺点是切块（chunking）会导致上下文连贯性丢失，尤其是中文注释嵌套英文代码的场景对 reranker 不友好；维护成本高（向量库、Embedding 模型、分片策略都需要管理）；首次构建索引较慢，小场景下显得过重。

文件上下文方式的优势是**保留原始结构**、无需向量数据库、对非技术用户友好（拖文件即可）。劣势是完全受大模型上下文窗口限制；当上下文被大量文档填满时，模型注意力分散，相当于考试时小抄带了整本书——定位慢、专注度低，导致推理成本高且回答质量下降；可解释性差，无法追踪"为什么用了这个文件"。

**选型建议**：给程序员交付时推荐 RAG，可以叠加高级策略（本文检索、文件溯源、元数据过滤）；给非技术用户交付时，文件上下文方式更简单直接。

## ML/深度学习与大模型的分工

大模型的兴起并不意味着机器学习和深度学习可以被抛弃。两者适合的场景不同：

传统 ML/深度学习擅长**感知和执行层**——计算机视觉（CV）、激光雷达定位、3D 扫描分析、工业机器人抓取轨迹等。这些场景对精度要求极高，大模型的幻觉问题在此无法接受。大模型擅长**语言理解和指令翻译层**——将自然语言转化为机器可执行的指令（"往前 5 米抓取物料放到传送带"），以及知识库问答和 Agent 反思推理。

一个合理的类比是：**大模型是大脑**，负责理解和决策；**ML/深度学习是手脚和感官**，负责感知和执行。两者互补，在工业自动化、具身智能等领域尤为明显。

大模型真正的核心价值场景只有两个：知识库问答（RAG）和工具调用型 Agent。RAG 和 Agent 这两种设计范式短期内不会过时，但技术迭代极快（上半年 RAG 火，下半年 Agentic RAG 出现），因此在大方向保持稳定的同时，仍需持续跟踪演进。

> [!example]- Answer Guide
> 
> #### Q1 — LangSmith vs LangFuse 分层逻辑
> 
> LangSmith 是 LangChain 官方服务，同时具备追踪和自动化测评能力，适合开发阶段跑性能评估；LangFuse 是完全开源的工具，只做 tracing 和告警，无法生成量化质量报告。分层使用的原因是：开发阶段需要完整评估能力，生产阶段需要控制商业授权成本和本地部署可控性，遇到问题再把数据带回本地用 LangSmith 分析。
> 
> #### Q2 — 70B 模型选型与记忆依赖
> 
> 参数过小（<70B）导致上下文理解不足、工具调用可靠性差；参数过大则 GPU 消耗过高、响应速度慢。记忆（短期/长期）可通过外部数据库实现，并不依赖模型本身，因此模型选型的核心判断标准是工具调用的准确性和参数映射的可靠性。
> 
> #### Q3 — 大脑与手脚类比分工
> 
> 大模型是"大脑"，负责将自然语言转化为机器可执行指令、知识库问答和 Agent 推理；传统 ML/深度学习是"手脚和感官"，擅长计算机视觉、激光雷达定位、工业机器人抓取等高精度感知执行任务——后者对幻觉零容忍，大模型无法替代。
