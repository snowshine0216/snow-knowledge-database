## 回顾：法律与金融场景

**法律场景两个死穴**：
1. **力度错配**：法条之间有逻辑依赖，暴力切片破坏完整逻辑单元；工程解法是层级感知切分 + 知识库谱
2. **关键字丢失**：向量数据库对数字和编号不敏感；工程解法是混合检索（Elasticsearch 关键词层 + 向量层）

**金融场景主要特点**：
- 大量跨页表格，存储形式可能是截图而非真实表格（导致表格撕裂）
- 最主要难点：**数值幻觉** + **沙箱执行**
  - 数值幻觉：用工具直接调用外部计算器（不靠大模型计算）
  - 沙箱执行：使用代码解释器工具或沙箱方式执行 Python

## 医疗场景：数据脏问题与本体映射

**为什么医疗数据"脏"**：

医疗领域存在三层语言不一致：

| 来源 | 表述 |
|------|------|
| 患者口语 | "嗓子疼"、"心窝子疼" |
| 病历书面语 | "咽痛"、"剑突下疼痛" |
| 知识库/医学书籍 | "急性咽炎"、"消化道疾病" |

三者说的是同一个事，但用词完全不同，直接放入向量库检索时，语义匹配失效。

**工程解法：本体映射（Ontology Mapping）**

在向量库之前增加一个 **NLP 自然语言处理清洗层**：

```
用户输入（口语）→ NLP 清洗层（实体识别）→ 标准术语 → 向量检索 → 结果
```

具体实现：
- 微调 BERT 模型做医学命名实体识别（NER）
- 构建口语 → 病历用语 → 医学标准术语的映射关系
- 识别完实体后，替换为标准术语，再从向量数据库检索

**示例**：
- "心窝子疼" → 实体识别 → "剑突下疼痛" → 向量检索 → 准确结果
- "嗓子疼" → "咽痛" → "急性咽炎" → 检索对应治疗方案

**为什么通用大模型（如豆包）在医疗自查场景准确率低**：大多数产品缺少这种实体映射层，直接用口语查询向量库，结果自然不准确。

## 医疗场景：护栏系统设计

医疗领域最大的风险是"随便回答"——AI 给出具体用药建议可能直接危害生命。因此需要构建**拒识系统（Guardrail System）**：

**整体架构**：

```
用户输入 → [前置护栏] → RAG + 大模型生成 → [后置校验] → 输出
```

---

## 医疗场景：前置护栏

使用**轻量级分类模型或意图识别模型**进行前置拦截，主要识别两类危险意图：

1. **危险自诊**：用户自行诊断严重疾病（如"我头疼，是不是 RCPO？"）
2. **处方药咨询**：用户询问处方药用法用量

触发以上任一情况，**直接返回固定化回复（不定化处）**：
- 内容为类似"仅供参考，请及时就医"的标准免责声明
- 不给出任何具体诊断或用药建议

**现实对比**：直接调用大模型 API（无产品护栏）时，模型可能直接回复"可以吃，计量如下..."，属于严重风险；成熟医疗 AI 产品必须有此前置护栏。

---

## 医疗场景：后置校验

大模型生成结果后，**不直接展示给用户**，先经过后置校验 Agent：

**多 Agent 架构**：

```
用户 → 意图分析 Agent → RAG + 生成 Agent → 批判/校验 Agent → 用户
                                                    ↓（拦截）
                                              固定回复（免责声明）
```

**后置校验 Agent 的典型 Prompt**：

```
请检查上述回答是否包含具体的用药建议，
如果是，请拦截并替换为固定的免责声明回复。
```

**历史教训**：早期模型设计不完善时，会出现"回答到一半突然截断"的问题——模型在流式输出过程中才反应过来"这个不能说"，导致用户体验极差。现代多 Agent 方案解决了这个问题：生成结果不直接流给用户，而是先给校验 Agent 判断，通过后再返回。

## 医疗场景：Human in the Loop

在互联网医院等场景中，AI 与医生协作的标准模式是**预问诊模式**：

```
AI（采集症状） → SOAP 结构化笔记 → 医生（审核 + 给出最终建议）
```

- **AI 负责**：采集症状、既往病史、过敏史，生成 SOAP 格式的结构化笔记
- **医生负责**：审核 AI 笔记，给出最终诊断和治疗建议

**类比**：与 AI 支付场景相同——
- AI 负责前端（推荐菜品、设置分量）
- 付款步骤必须人工确认（Human in the loop）
- 确认后流转回自动化系统

**医疗 AI 的本质**：类似推荐系统，理解用户意图并推荐（诊断），但最终决策权和收口权留给医生。

## 三个垂直领域的能力总结

| 领域 | 核心难点 | 专属工程能力 |
|------|----------|--------------|
| 法律 | 力度错配、关键字丢失 | 混合检索（BM25 + 向量）+ 知识图谱 |
| 金融 | 非结构化文档、数值幻觉 | OCR 版面分析 + Python 沙箱执行 |
| 医疗 | 术语不统一（数据脏）+ 伦理风险 | 本体映射（NLP 实体识别）+ 护栏系统（前置+后置）|

**通用底座**（三个领域共享）：
- RAG 检索
- 多 Agent（意图分析、生成、校验）
- 工具调用（工具调用是解决计算幻觉的关键）
- Human in the loop（高风险决策必须人工收口）

**从业者建议**：
- 分析自己所在行业，对照三个领域的解法，找出需要额外补充的能力
- 丢掉具体技术细节，用领域专家眼光看 AI 需要哪些能力
- 有专业背景的行业专家参与是做好垂直 AI 产品的前提
