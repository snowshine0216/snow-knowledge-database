## 1. 模块概述：多模态的三类场景

本讲是模块7的专题内容，聚焦 Agent 在多模态任务中的实际应用。课程介绍了三个核心方向：

1. **文字 ↔ 图像语义搜索**：使用 CLIP 模型 + 向量数据库，实现文字查图 / 图查图
2. **RPA + 大模型**：借助 PyAutoGUI 或八爪鱼等工具，让 Agent 操控桌面和网页，完成屏幕截图识别与自动化操作
3. **移动端小模型优化**（下一讲）：模型量化、剪枝、蒸馏，应对端侧部署需求

本讲重点讲解前两个场景中的图像-文字跨模态能力。

## 2. 文字 → 图像：CLIP + Milvus 图像语义搜索

### 场景需求

当系统中存有大量图片，需要用户通过**自然语言**来搜索图像时（如"一个孩子在雨天打着红伞蹦跳地走过水坑"），传统 OCR 或关键字匹配无法完成跨模态语义匹配，需要将文字与图像嵌入到同一向量空间中进行相似度检索。

### 整体流程

```
[离线索引阶段]
图片库 → CLIP 图像编码器 → 图像向量 → 存入 Milvus

[在线搜索阶段]
用户文字查询 → CLIP 文本编码器 → 文字向量 → Milvus 向量相似度搜索 → 返回 Top-K 图片
```

### 关键组件

| 组件 | 作用 |
|------|------|
| CLIP 模型（ViT-B/32） | 将文字和图像编码到同一向量空间 |
| Milvus 向量数据库 | 存储图像向量，支持高效相似度检索 |
| Image Processor | 将搜索结果的多张图片合并为一张展示图 |
| Logger / 装饰器 | 性能计时（`@log_execution_time`）与业务日志 |

---

## 3. CLIP 模型原理：对比学习与联合嵌入

CLIP（Contrastive Language–Image Pre-training）的核心是**对比学习**：

- 训练时，将图文对（如"孩子红伞的图片"与"孩子红伞"的文字）同时送入模型
- 若图文语义相近，模型直接拟合（正例）；若语义较远，调整模型参数使其靠近（对比损失）
- 训练结果：语义相近的图文在向量空间中距离很近

**工程使用时无需自己训练**，直接使用 OpenAI 发布的预训练权重（如 `ViT-B/32`）即可。更大的模型（如 `ViT-L/14`）精度更高但显存占用更大。

## 4. Milvus 向量数据库：安装与操作

### 三种运行模式

| 模式 | 特点 |
|------|------|
| **Milvus Lite** | 轻量级，适合本地测试和数据验证 |
| **Standalone** | 独立单节点运行，监听端口 `19530`（gRPC）和 `9091`（HTTP） |
| **Cluster（Docker Compose）** | 生产模式，包含主节点、从节点和配置同步节点，启动后有 3 个容器 |

### 核心操作

```python
# MilvusManager 封装的主要能力
insert_data()        # 插入图像向量
create_index()       # 建立索引
search()             # 向量相似度搜索（返回 Top-K 结果）
```

索引（indexing）操作耗时较长，应在用户访问量低峰期执行，与查询流程分离。已存在的图像无需重复索引（去重检查）。

## 5. 代码架构：模块化 Python 项目实践

讲师通过该项目演示了 Python 模块化的工程实践：

### 模块拆分方式

| 文件 | 职责 |
|------|------|
| `clip_image_search.py` | 主程序：工作流编排（setup → index → search → display） |
| `clip_encoder.py` | CLIP 模型封装：文字/图像 → 向量 |
| `milvus_manager.py` | Milvus 向量数据库操作封装 |
| `image_processor.py` | 图像处理：多图合并展示 |
| `logging_config.py` | 日志系统：统一配置，支持时间戳、日志级别、调用位置 |

### 日志系统设计

推荐维护两类日志：
1. **耗时日志**：通过 `@log_execution_time` 装饰器记录各函数执行时间，定位性能瓶颈
2. **业务日志**：按步骤追踪程序执行流程，辅助故障排查

日志在类的 `__init__` 方法中初始化，覆盖整个对象生命周期。

### 代码复用原则

核心洞察：**代码库应越写越简单**。通过将 Milvus、CLIP、日志能力封装成独立工具模块，后续项目直接复用，无需重写。使用大模型分析已有代码的架构技巧，再让其用相同风格优化新代码。

## 6. 图像 → 文字：视觉理解模型

### 为何不用 CLIP 做反向检索

CLIP + Milvus 方案仅支持"文字查图"。若输入图片并需要生成文字描述或推理结果，该路径无法直接反向使用，需改用多模态大模型。

### Qwen-VL 视觉理解能力

以 **Qwen-VL-Max**（千问VL）为代表的多模态大模型支持：

| 能力 | 描述 |
|------|------|
| 视觉理解（图像问答） | 描述图像中的对象、背景、光线、颜色、空间关系 |
| 视觉推理 | 识别图像中人物动作、场景语义、异常状态 |
| 文字提取 | 识别图像中的文字（含复杂背景） |

演示：给模型输入一张香蕉图片，模型能识别"手拿着半剥香蕉皮的香蕉"，并描述房间光线、颜色背景等细节。

## 7. 视觉理解 vs. 传统 OCR 的核心区别

| 对比维度 | 传统 OCR | 视觉理解模型 |
|----------|----------|-------------|
| 识别依据 | 亮度、颜色、边缘特征 | 语义理解（理解图像含义） |
| 过曝/低对比场景 | 无法识别 | 可通过场景推理补全 |
| 物体识别 | 需要精确标注 | 直接理解物体、动作、关系 |
| 成本 | 低 | **较高** |

### 典型工程场景

1. **工地安全监控**：判断电焊工是否在正常作业，还是在抽烟偷懒
   - 传统 OCR：被过曝的焊接亮光干扰
   - 视觉理解：识别人物姿态和行为语义

2. **低空经济/无人机巡检**：识别山火还是汽车尾灯
   - 传统 OCR：亮度相近导致误识别
   - 视觉理解：识别整体场景中"汽车"与"火焰"的语义差异

3. **内容安全审核**：识别图像/视频中嵌入的违规文字
   - 变形字、连线干扰字：OCR 无法拆分单个字符
   - 视频帧逐帧分析：将视频拆为独立帧，对每帧用视觉模型识别
   - 视觉理解可识别扭曲/嵌入文字并判断违规内容

## Key Takeaways

1. **多模态 Agent 的两条主线**：文字 → 图像用 CLIP + 向量数据库；图像 → 文字用多模态大模型（Qwen-VL / GPT-4V / Claude 等）。

2. **CLIP 是跨模态搜索的关键桥梁**：将文字和图像映射到同一向量空间，通过向量相似度实现语义匹配，无需手工标注。

3. **Milvus 是图像向量存储的首选**：支持三种部署模式，核心操作为 insert/create_index/search，索引与查询应分离运行。

4. **视觉理解 > 传统 OCR 的根本原因**：OCR 依赖像素特征，视觉大模型理解语义，适合处理复杂场景、异常亮度、嵌入文字等 OCR 无法解决的问题。

5. **视觉模型很贵，要酌情使用**：仅在 OCR 无法解决且业务价值足够高时才引入视觉理解模型。

6. **工程代码复用原则**：将通用能力（向量数据库、CLIP 编码器、日志）封装为独立模块，用大模型辅助架构调整，使代码越写越简单。
