## 实验管理的意义

大模型微调本质上是无数次实验。核心挑战在于：**我们不知道哪一次实验结果是最优的**（就像买房时不知道哪天是价格最低点）。

**解决方案：检查点（Checkpoint）机制**

- 每训练 N 步保存一次检查点（类似游戏存档）
- 训练结束后，从所有检查点中找出验证集 Loss 最低的一个
- 使用该最佳检查点作为最终模型

实验管理的三个核心目标：

```
问题 → 证据 → 决策  （铁三角）
```

| 目标 | 说明 |
|------|------|
| 快速定位最优配置 | 减少无效实验次数 |
| 避免上线灾难 | 在部署前发现配置问题 |
| 说服决策者 | 用业务语言（非技术参数）汇报改进效果 |

**业务语言示例**：不要说"我把 LoRA rank 从 16 改成 8"，而要说"方言识别率从 58% 提升到 82%，用户投诉率下降 30%"。

## 核心超参数详解

### 1. 损失函数（Loss Function）vs 代价函数（Cost Function）

| 概念 | 含义 | 别名 |
|------|------|------|
| Loss（损失） | 单个样本的预测误差 | 损失函数 |
| Cost（代价） | 全体训练集的平均误差（MSE：均方误差） | 代价函数 |

**实践说明**：论文和工程代码中这两个概念**经常混用**，统一理解为"损失函数"即可。训练时看到 `loss` 关键词表示的就是这两个概念之一。

目标：找到损失函数曲线的**最低点**（最优解）。

---

### 2. 学习率（Learning Rate）

**定义**：参数更新时的步长系数，控制每次梯度下降的幅度。

**常用值**：`1e-4` 或 `1e-5`

| 学习率设置 | 效果 | 问题 |
|------------|------|------|
| 过高（如 0.1） | 快速更新 | 在最优解附近震荡，永远找不到最低点（跳过最优解） |
| 过低（如 1e-7） | 稳定收敛 | 训练时间和 GPU 开销极大 |
| 适中（1e-4~1e-5） | 快速且稳定收敛 | — |

**动态学习率策略**：
- **线性衰减**：训练过程中学习率逐渐减小，使模型在接近最优解时步长更小，表现为日志中学习率持续下降（如从 `2e-5` 缓慢降至 `1e-6`）。
- **早停法（Early Stopping）**：当验证集 Loss 不再下降甚至开始上升时，自动停止训练。

---

### 3. Batch Size（批次大小）

**定义**：每次梯度更新时使用的样本数量。

| Batch Size | 优点 | 缺点 |
|------------|------|------|
| 大 | 梯度更稳定，训练更高效 | 显存/内存消耗大（容易 OOM），泛化能力下降（过拟合） |
| 小 | 显存友好，有正则化效果 | 梯度噪声大，容易跳出最优解，训练速度慢，收敛不稳定 |

**实践**：遇到 OOM（Out of Memory）时，调小 batch size。

---

### 4. eval_steps（评估步数间隔）

**定义**：每训练多少步在验证集上计算一次 Loss 并保存检查点。

- 设置**过小**：频繁评估带来额外开销。
- 设置**过大**：可能错过最优检查点（最优点恰好出现在两次评估之间）。

**建议**：约每 **50 步** 保存一次检查点（具体视总训练步数而定）。

---

### 5. num_train_epochs（训练轮数 / Epoch）

**定义**：模型对整个训练集完整遍历一次称为 1 个 Epoch。

| Epoch 设置 | 问题 | 表现 |
|------------|------|------|
| 过小 | 欠拟合（Underfitting） | 模型还没学完就停止，考试时不会答题 |
| 过大 | 过拟合（Overfitting） | 只会背训练集答案，泛化能力差 |

**早停法**：不预设固定 Epoch，而是监控验证集 Loss，不再提升时自动终止。

## 五次实验：从失败到收敛

以下为一组典型的自我认知微调实验（数据集约 100 条，基础参数：学习率 `1e-4`，rank=4，epoch=1）：

### 实验一：学习率过高

| 参数 | 值 |
|------|-----|
| 学习率 | 0.1 |
| LoRA rank | 4 |
| Epoch | 1 |

**结果**：训练 Loss 和验证 Loss 均**持续上升**。

**原因**：学习率设置过高，模型在参数空间中反复震荡，无法找到最优解。

**解决**：大幅降低学习率。

---

### 实验二：欠拟合（Underfitting）

| 参数 | 值 |
|------|-----|
| 学习率 | 0.0005 |
| LoRA rank | 4 |
| Epoch | 1 |

**结果**：训练 Loss 开始下降，验证 Loss **先降后升**。

**原因**：训练轮数太少，模型还没有充分学习数据集就停止了（欠拟合）。

**解决**：增加 Epoch 数量，让模型继续训练（**不要改其他参数**）。

---

### 实验三：过拟合（Overfitting）

| 参数 | 值 |
|------|-----|
| 学习率 | 0.0005 |
| LoRA rank | 4 |
| Epoch | 50 |

**结果**：训练 Loss 快速下降后缓慢降低（弧线），验证 Loss **先降后升**，且上升更为明显。

**原因**：Epoch 过多而数据量不足（仅 100 条），模型"背下了"训练集答案，泛化能力差。

**解决**：增加训练数据量。

---

### 实验四：接近收敛（Underfitting，数据量不足+rank偏低）

| 参数 | 值 |
|------|-----|
| 学习率 | 0.0005 |
| LoRA rank | **8**（从 4 增大） |
| Epoch | 50 |
| 数据量 | **1000+ 条**（从 100 增加） |

**结果**：训练 Loss 和验证 Loss 均下降，**仍有轻微欠拟合**，但已经非常接近成功。

**解决**：再增加 Epoch。

---

### 实验五：成功收敛

| 参数 | 值 |
|------|-----|
| 学习率 | 0.0005 |
| LoRA rank | 8 |
| Epoch | **15** |
| 数据量 | 1000+ 条 |

**结果**：训练 Loss 快速下降后缓慢趋于平稳；验证 Loss 与训练 Loss 曲线**几乎吻合**，最终略微上升但整体稳定。

**判断标准**：`训练 Loss ≈ 验证 Loss ≈ 接近 0` → **训练成功**。

```
实验结果对比：
- 实验一：Loss ↑↑  → 学习率过高
- 实验二：验证 Loss 先↓后↑ → 欠拟合（训练不足）
- 实验三：验证 Loss 快速↑  → 过拟合（数据不足）
- 实验四：几乎收敛          → 欠拟合（轻度）
- 实验五：训练验证曲线吻合  → 收敛成功 ✓
```

## LoRA 合并与部署策略

### 不合并（Adapter 模式）

LoRA checkpoints（约 200–300 MB）与基础模型（如 Qwen2.5-7B 约 14 GB）分开存储。

**推理方式**：

```bash
swift infer \
  --model_type qwen2_5-7b-instruct \
  --adapters output/<timestamp>/checkpoint-94 \
  --stream true \
  --max_new_tokens 2048
```

**缺点**：推理时需额外计算步骤，增加约 **5–10% 延迟**。

**适用场景**：开发阶段、需要频繁切换多个 LoRA 任务（一个基模型 + 多个 LoRA）。

### 合并模式（Merge & Deploy）

将 LoRA 参数合并进基础模型，生成完整模型文件。

**合并 + 推理命令（Swift）**：

```bash
swift infer \
  --model_type qwen2_5-7b-instruct \
  --adapters output/<timestamp>/checkpoint-94 \
  --merge_lora true \
  --infer_backend vllm
```

**优点**：
- 推理无额外开销，性能与微调前几乎相同
- 部署更简单，兼容性更好（可用 vLLM 等框架）

**缺点**：
- 无法快速切换任务
- 存储空间增大（完整模型文件）

**选择建议**：

| 场景 | 推荐 |
|------|------|
| 开发/调试阶段 | Adapter 模式（分离） |
| 私有化单任务部署 | 合并模式 |
| 多租户/多任务 SaaS | Adapter 模式（灵活切换） |
| 边缘设备 | 合并模式 |

### 蓝绿发布策略

- 将合并后的模型版本部署上线（绿）
- 同时在本地继续迭代新版本的 Adapter（蓝）
- 验证通过后合并新版本并切换上线

## 模型评估：考试类比

微调效果评估有两种方式：

### 方式一：观察 Loss 曲线

通过检查点的训练/验证 Loss 曲线判断训练状态（参见上文五次实验）。

### 方式二：直接对话测试

让模型执行具体任务，与基准测试结果对比。

**示例：IoT 意图识别测试**

| 模型状态 | 输入 | 输出 |
|----------|------|------|
| 未微调 | "地上有点脏" | 解释地板脏了会怎样，给出建议 |
| 微调成功 | "地上有点脏" | 调用扫地机器人，记录地面清洁需求 |
| 欠拟合 | "灯光有点暗" | 给出调节建议，不执行指令 |
| 微调成功 | "灯光有点暗" | 直接调节灯光亮度 |

**最佳检查点的确认**：
- 训练完成后日志会输出 `best_model_checkpoint`
- 优先使用该检查点，而非最后一步（`last`）
- 若没有输出 `best`，检查验证集 Loss 最低的那一步

## 实验管理最佳实践

1. **只关注关键参数**（不要试图搞定所有参数）：
   - **学习率**（`learning_rate`）：最重要
   - **LoRA rank**（`lora_rank`）：通常设为 8
   - **训练轮数**（`num_train_epochs`）：次重要
   - 其他参数保持默认即可

2. **记录决策依据**：每次实验记录参数设置 + Loss 曲线 + 业务表现，建立实验日志（类似代码版本管理的 commit message）。

3. **分层诊断**：
   - Loss 持续上升 → 检查学习率（过高）
   - 训练 Loss ↓、验证 Loss 先↓后↑ → 增加 Epoch
   - 验证 Loss 比训练 Loss 高很多 → 增加数据量或降低 Epoch
   - OOM → 减小 batch size 或 max_length

4. **数据量经验**：
   - 意图明确的分类任务：约 **1000 条**
   - 数据少于 100 条：容易欠拟合，尤其在多 Epoch 时

5. **用业务指标沟通**：向团队或管理层汇报时，翻译为业务语言（如准确率、延迟、用户投诉率等）。
