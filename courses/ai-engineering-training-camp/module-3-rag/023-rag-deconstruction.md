## RAG 的本质与定义

RAG（Retrieval-Augmented Generation）的本质是：**在用户提问之前，向提示词中插入一段从知识库检索出的参考内容，告诉大模型基于这段内容来回答问题**。

讲师定义：RAG 是结合传统信息检索技术（搜索与推荐系统）和大模型生成技术的一种**处理方法（解决方案）**，而非单一技术。把 RAG 理解为解决方案比理解为框架更有利于学习——任何框架都可以替换，但 RAG 的核心逻辑不变。

大模型产生幻觉的四大原因：
1. 训练语料本身包含不正确或已过时的事实
2. 领域知识不足（训练数据不够）
3. 过度生成（指令集驱动模型输出看似合理但不真实的内容）
4. 模糊输入（用户问题不清晰，上下文冲突）

RAG 的核心价值：消除幻觉对用户信任的损害，尤其在企业交付场景中。

## 为什么不能用长上下文代替 RAG

即使上下文窗口足够大（如 128K+），把所有文档直接丢给大模型也存在两个根本问题：

1. **注意力稀释**：文档过长时，模型可能错过正确答案，因为注意力被无关内容分散
2. **语义冲突**：不同文档之间存在相互矛盾的信息（例如安全部门和研发部门对同一事项的"注意事项"内容相反），大模型无法辨别哪个更权威

此外，公司内部知识、私有数据、实时信息天然无法通过公开训练获得，只能通过 RAG 注入。

## RAG vs 微调

| 维度 | RAG | 微调 |
|------|-----|------|
| 可解释性 | 高（可追溯召回片段来源） | 低（黑盒，难以诊断效果差的原因） |
| 知识更新 | 更新知识库即可，近实时 | 需要重新训练，成本高 |
| 外部知识需求 | 高外部依赖时首选 | 主要用于调整输出风格/格式 |
| 错误调试 | 可分层排查（检索 vs 生成） | 只能靠调参试错 |

微调更适合改变模型的输出方式（表达风格），而非注入知识。RAG 适合外部知识依赖度高、数据实时性要求强的场景。

## RAG 的五阶段框架

主流 RAG 框架（LlamaIndex、LangChain 等）统一将流程划分为五个阶段：

### 1. 加载（Loading）
从 PDF、Word、网站、数据库、API 等多种来源加载原始数据，统一转化为**文档（Document）**格式，并附加**元数据（Metadata）**：
- 文件名称、创建时间
- 文档摘要、章节信息
- 来源 URL（支持文本溯源）

不同文件格式的解析挑战：
- **Word**：XML 格式，可精确区分标题、正文、页眉页脚；内嵌图片需交给多模态模型单独解析
- **PDF**：无结构化标签，扫描件需 OCR；需针对性解析器，甚至多次解析

### 2. 索引（Indexing）
将文档切分为**分块（Chunk）**并存储：
- 分块是 RAG 必须具备的能力——否则整篇文档上传大模型即可，无需 RAG
- 分块后进行**向量嵌入（Embedding）**，转化为数值向量
- 存储方式：向量存储（用于语义相似度检索）和原文存储（用于关键词检索）

### 3. 检索（Retrieval）
两种检索方式往往并存：
- **向量检索**：基于语义相似度，适合模糊查询
- **关键词检索（BM25）**：基于实体匹配，精确度高

提取用户问题中的实体后再检索效果更好。

### 4. 查询/重排（Query / Reranking）
- 对两路检索结果合并后，按相似度排序
- 可放置一个**重排序模型（Reranker）**，基于用户原始问题对片段重新排序，提升精度

### 5. 生成（Generation）
将召回片段 + 用户问题 + 提示词模板组合，送入大模型生成最终回答。

## 离线索引 vs 在线查询

```
离线流程（Offline Pipeline）
原始文档 → 解析 → 分块 → 嵌入 → 存入向量库

在线流程（Online Pipeline）
用户提问 → [问题优化] → 检索 → [重排] → 提示词组合 → 大模型生成 → 返回答案
```

离线索引可以定时任务增量更新；演示系统可支持实时上传后立即查询（但文档量不能太大，否则 I/O 成本极高）。

## 检索前优化

用户问题会被使用**两次**：
1. 检索阶段：用于查询向量库
2. 生成阶段：作为大模型输入的一部分

因此，检索前对问题的改写**不能影响最终传给大模型的原始问题语义**。

四种常见的问题改写策略：
1. **补充上下文**：用户角色（开发工程师 → 开发工程师的注意事项）
2. **扩写/细化**：将模糊问题拆分为更具体的子问题
3. **反问澄清**：请用户提供更多信息
4. **步骤拆分**：将复杂问题拆解为多步查询（Multi-step Query Engine）

**HyDE（假设文档嵌入）**：先让大模型基于用户问题生成一个假想答案（不参考知识库），再用这个假想答案去检索真实文档，最后基于真实文档生成回答。适用于用户问题极其模糊时。

**意图识别 vs 问题改写**：二者不冲突，意图识别是路由（决定走哪条 RAG 分支），问题改写是在确定路由后对检索 query 的优化。将 RAG 整体作为一个 agent 工具，意图识别放在 RAG 之外，形成 **Agentic RAG**。

## 检索后优化

- 对两路检索结果（向量 + 关键词）进行合并和去重
- 通过重排序模型重新按相关性排列
- 可进行内容总结、融合，减少噪声

## 朴素 RAG → 进阶 RAG → 模块化 RAG

```
提示词 → 朴素 RAG（Simple RAG）→ 进阶 RAG（Advanced RAG）→ 模块化 RAG → Agentic RAG
```

- **朴素 RAG**：直接检索 + 生成，漏洞多，目前几乎无人在生产中使用
- **进阶 RAG**：在检索前（问题改写）和检索后（重排）增强
- **模块化 RAG**：强调整体架构可插拔，各组件独立替换（LlamaIndex、Dify 均属此类）
- **Agentic RAG**：RAG 作为 agent 工具，配合意图识别和多步推理

## RAG 评估指标

### 纯检索指标
| 指标 | 含义 |
|------|------|
| 精确率（Precision） | 检索到的相关文档 / 检索到的所有文档（信噪比） |
| 召回率（Recall） | 检索到的相关文档 / 所有相关文档（完整性） |
| F1 | 精确率与召回率的调和平均值 |

### 检索 + 重排组合指标
| 指标 | 含义 |
|------|------|
| MRR（平均倒数排名） | 最相关文档在结果列表中排名的倒数平均值 |
| MAP（平均精确率均值） | 每个 query 的平均精确率再取均值 |
| NDCG（归一化折扣累积增益） | 对排名越靠前的相关文档给予更高权重 |

### 生成评估指标
1. **上下文相关性**：检索到的文档是否与 query 相关
2. **答案忠实性**：生成答案是否基于提供的上下文（而非大模型"凭空捏造"）
3. **答案相关性**：最终答案是否回答了用户问题

注意：大模型自动评估 RAG 质量的准确率目前仍很低（GPT-4 级模型约 40-79%），建议人工评估。

> [!example]- Answer Guide
> 
> #### Q1 — RAG Five-Stage Framework
> 
> 五阶段分别是：
> 
> - **加载**：多源数据统一转为带元数据的 Document
> - **索引**：分块 + 向量嵌入 + 存储
> - **检索**：向量语义检索 + BM25 关键词检索双路并行
> - **查询/重排**：合并去重 + 重排序模型按相关性重排
> - **生成**：召回片段 + 用户问题 + 提示词模板送入大模型
> 
> **离线流程**涵盖前两阶段（解析→分块→嵌入→入库）；**在线流程**涵盖后三阶段（检索→重排→生成）。
> 
> #### Q2 — HyDE Hypothetical Document Embeddings
> 
> HyDE 的流程：先让大模型基于用户问题生成一个**假想答案**（不参考知识库），再用这个假想答案去检索真实文档，最后基于真实文档生成最终回答。
> 
> 它解决的问题是用户问题极其模糊、难以直接匹配知识库的场景。
> 
> 与普通问题改写的区别：它是用"生成的答案"而非"改写的问题"来驱动检索。
> 
> #### Q3 — RAG vs Fine-tuning Comparison
> 
> | 维度 | RAG | 微调 |
> | --- | --- | --- |
> | **可解释性** | 高——可追溯召回片段来源 | 黑盒 |
> | **知识更新** | 只需更新知识库（近实时） | 重新训练，成本高 |
> | **错误调试** | 可分层排查（检索 vs 生成） | 只能靠调参试错 |
> 
> RAG 适合外部知识依赖度高、实时性强的场景；微调更适合调整模型的输出风格和格式，而非注入知识。
