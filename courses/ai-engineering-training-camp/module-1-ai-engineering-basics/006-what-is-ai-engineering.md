## AI 工程化的定义：从简单到复杂

AI 工程化不是凭空出现的，而是因为**代码越写越复杂**。一段 20 行的 Hello World 式 OpenAI API 调用根本不需要工程化——没有维护性问题、复用问题、扩展性问题。但当代码规模扩大，问题就会自然出现：

- **Prompt 管理**：多套提示词框架如何命名、索引、按需调用？自然联想到哈希表（名字 → 值）和策略模式（按场景切换不同优化策略——HyDE 想象答案、对话历史补全、Few-Shot 举例）。
- **模型网关**：业务增长后一个模型无法承载请求，需要负载均衡；硬件代际差异导致大小模型共存；不同任务需要不同参数规模的模型。这就需要大模型网关，并考虑模型存活探活、服务发现与注册。

这些问题都来自复杂化之后的自然涌现，正如数据库引入缓存后才会出现缓存穿透和雪崩——没用过就不会考虑。工程化的核心是：在代码复杂度达到临界点之后，用**分层架构**来做关注点分离。

## AI Agent 的八层逻辑架构

研究者 Greg 提出的八层逻辑模型，是概念层级（类比 OSI 七层），用于建立思考框架而非直接编码：

| 层级    | 名称         | 核心内容                                                          |
| ----- | ---------- | ------------------------------------------------------------- |
| 第 1 层 | 基础设施层      | 核心计算、网络、存储、云、边缘计算。**下层制约上层**——算力不足时要往下层扩充硬件，而非在模型层做参数优化       |
| 第 2 层 | AI 智能体互联网层 | 跨网络通信（A2A 协议）、分布式协调（Ray 框架）、**Agent 发现与注册**、信任与名誉系统（多供应商互信评分） |
| 第 3 层 | 协议层        | A2A、ACP、AGP 等通信协议；数字签名、访问控制、加密隧道等安全机制；MCP（讲师认为 MCP 应归于更底层）    |
| 第 4 层 | 工具增强与扩展层   | 向量数据库、RAG、Function Calling、Python 虚拟运行环境等外部能力扩展               |
| 第 5 层 | 认知与推理层     | 逻辑推理引擎、多 Agent 战略规划、AutoGen 等角色框架。**工程师主要在此层和第 4 层"堆屎山"**     |
| 第 6 层 | 记忆与个性化层    | 用户行为偏好分析、响应定制                                                 |
| 第 7 层 | 特定领域智能（垂类） | 行业专用能力                                                        |
| 第 8 层 | 运维与治理层     | 合规监管、安全风险管理、策略执行。偏学术，主要是**发 paper 的层**                        |

**层级思维的价值**：遇到问题时先定位层级，再确定解决方案。算力不足 → 第 1 层扩硬件，而非第 5 层调参。负载均衡 → 第 2 层，而非业务逻辑层。

## 落地六层技术栈

八层逻辑架构类比 OSI，实际落地时压缩为六层技术栈，类比 TCP/IP：

1. **基础设施层**（Infrastructure）
2. **架构层**（Architecture）
3. **基础模型层**（Foundation Model）
4. **大语言模型层**（LLM）
5. **API 层**
6. **应用层**（Application）

建议作业：将自己熟悉的技术（LangChain、Dify、向量数据库、Redis 缓存等）按此六层摆放，理解层间通信方式、下层对上层的支撑、上层对下层的封装。这比背概念更能建立架构直觉。

> [!info]+ 💡 Explanation - 分层为什么不是"造名词"？
> 
> 理解你的困惑。技术分层（Layering）在工程师眼中往往显得冗余，甚至像在"造名词"。但如果你经历过因为模型升级（比如从 GPT-3.5 换到 Claude 3）而导致整个业务代码重写的痛苦，你就会明白分层的苦心。
> 
> 分层的核心目的只有一个：**解耦（Decoupling）**。它让你在更换底层"零件"时，不需要拆掉整座"大楼"。
> 
> ### 为什么需要分层？（分层的哲学）
> 
> 在没有分层的情况下，你的代码是**"面条式"**的：API 调用、Prompt 拼接、数据库检索全部搅在一起。
> - **下层对上层的支撑：** 下层提供稳定、标准的能力（如：向量数据库提供检索能力）。
> - **上层对下层的封装：** 上层屏蔽下层的复杂性（如：LangChain 让你不用管向量数据库用的是 Milvus 还是 Pinecone，它只给你一个 `similarity_search` 接口）。
> 
> ### 六层技术栈详细拆解
> 
> **1. 基础设施层 (Infrastructure)**
> - **类比：** 发电厂与机房。
> - **核心：** GPU (A100/H100)、CUDA 环境、云服务（AWS/阿里云）、K8s 容器编排。
> - **你的技术位置：** 如果你是在本地跑模型，**Ollama** 或 **vLLM** 勉强算在这一层（提供算力调度）。
> 
> **2. 架构层 (Architecture / Data & Memory)**
> - **类比：** 图书馆与缓存间。
> - **核心：** 解决大模型的"长短期记忆"问题。
> - **你的技术位置：**
>   - **向量数据库 (Milvus, Pinecone, Weaviate)：** 存放知识库，支撑 RAG。
>   - **Redis：** 存放对话上下文（Session），实现高速缓存。
>   - **PostgreSQL：** 存放结构化用户信息。
> 
> **3. 基础模型层 (Foundation Model)**
> - **类比：** 一个博学但没有经过职业训练的大脑。
> - **核心：** 原始的权重文件。
> - **你的技术位置：** **Llama 3**、**GPT-4 (Base)**、**Mistral**。这一层只负责最基本的预测下一个词的能力。
> 
> **4. 大语言模型层 (LLM / Alignment & Logic)**
> - **类比：** 经过职业培训的专家（如医生、律师）。
> - **核心：** 这一层是在基础模型之上，通过 **SFT (精调)** 或 **复杂的 Prompt Engineering** 形成的逻辑核心。
> - **意义：** 它决定了模型是扮演一个"翻译官"还是"代码助手"。
> 
> **5. API / 框架层 (The Orchestrator)**
> - **类比：** 翻译官与项目经理。
> - **核心：** 负责把模型、数据库、工具连起来。
> - **你的技术位置：**
>   - **LangChain / LlamaIndex：** 它们是这一层的霸主。它们定义了"如何检索、如何思考、如何行动"的标准流。
>   - **API Gateways：** 统一调度不同厂家的 API。
> 
> **6. 应用层 (Application / UI)**
> - **类比：** 手机 App 或网页。
> - **核心：** 面向最终用户，提供交互界面。
> - **你的技术位置：**
>   - **Dify / Coze：** 它们跨越了 API 层和应用层。Dify 既帮你管理了 Workflow（API 层），又提供了前端 Chatbot（应用层）。
> 
> ### 将你熟悉的技术填入"作业表"
> 
> | 层级 | 代表技术 | 它的作用 |
> | :--- | :--- | :--- |
> | **应用层** | **Dify** / Web 界面 | 用户在这里说话，看到反馈。 |
> | **API 层** | **LangChain** | 组织逻辑，把 Prompt 发给模型，把问题发给向量数据库。 |
> | **LLM 层** | **GPT-4o** / **Claude 3.5** | 接收指令，思考并生成文本逻辑。 |
> | **基础模型层** | **Llama 3 (Original)** | 支撑 LLM 的原始智能。 |
> | **架构层** | **Vector DB (Milvus)** / **Redis** | 提供 RAG 检索和对话记忆存储。 |
> | **基础设施层** | **NVIDIA GPU** / **vLLM** | 负责最底层的矩阵运算。 |
> 
> ### 你的"四层简化版" vs 六层
> 
> 你提出的简化逻辑：**模型 -> 工具 -> 框架 -> 调用**。
> 这非常适合**开发者视角**，但六层模型更适合**架构视角**。
> 
> - **模型层：** 对应 3 和 4。它区分了"裸模型"和"业务逻辑模型"。
> - **工具类层：** 对应 2。比如向量数据库其实是模型的"外挂工具"。
> - **框架层：** 对应 5。比如 LangChain。
> - **调用层：** 对应 6。
> 
> **为什么要用六层？**
> 因为在实际落地时，**"架构层" (数据存储)** 的设计往往比模型本身更复杂。如果把向量数据库只看作一个"工具"，你可能会忽视数据同步、索引优化、多路召回等架构问题。
> 
> ### 总结
> 
> 分层不是为了把简单问题复杂化，而是为了让你在 **Dify (应用)** 里点点鼠标换掉 **LangChain (框架)** 逻辑，或者在 **LangChain** 里一行代码把 **OpenAI (LLM)** 换成 **DeepSeek (LLM)**，而不需要去动底层的 **Milvus (架构)** 数据。
> 
> 你目前正在处理的落地场景中，哪一层让你觉得最难调优？我们可以深入聊聊那一层。

> [!question]- 📋 面试题 (Interview Follow-up)
> 
> **题目 1：** 如果你要把一个在 OpenAI 上跑的 Chatbot 迁移到本地的 Llama 模型，而且要求不改动业务代码，你该如何设计架构？
> 
> **题目 2：** 你的向量数据库检索有时候不准确。你会在哪一层来优化？为什么不在其他层优化？
> 
> **题目 3：** Dify、LangChain、向量数据库，这三个工具你会怎么安放在六层里？它们的关系是什么？

> [!example]- 💡 答案指南 (Answer Guide)
> 
> **题目 1 - 引导答案思路：**
> - 不改动业务代码 → 说明你的**应用层**（第 6 层）不能变。
> - 模型源从 OpenAI 换成本地 → 说明**大语言模型层**（第 4 层）和**基础设施层**（第 1 层）会变。
> - 核心做法：把**大语言模型层**的调用封装在一个统一接口（比如 LangChain 的 `LLMChain` 或自己写的 `ModelInterface`），这样应用层永远面对的是这个接口，而不是具体的模型实现。
> - 架构答案的精妙：通过**第 5 层（API 层）的适配**，使得**第 6 层（应用）无感知地切换第 4 层（模型）**。
> 
> ---
> 
> **题目 2 - 引导答案思路：**
> - 这个问题的关键是区分"问题的根源"和"问题的症状"。
> - 症状：Chatbot 给了错误答案 → 表现在**应用层**（第 6 层）。
> - 根源：向量检索的质量 → 根源在**架构层**（第 2 层）。
> - 可以在的优化点：
>   - **第 2 层（架构）：** 调整向量数据库的索引参数、调整 embedding 模型（从 OpenAI 的 embedding 换到 BGE）、多路召回策略。
>   - **第 4 层（LLM）：** 用 Prompt Engineering 让模型在输出时"承认不知道"而不是凭空编造。
>   - **不应该只在第 6 层（应用）解决：** 比如说，只在前端加一个"确认信息准确性"的按钮——这只是掩盖问题，不是解决问题。
> - 工程化思维：**先在第 2 层解决数据质量问题，再在第 4 层用 Prompt 兜底。**
> 
> ---
> 
> **题目 3 - 引导答案思路：**
> - **LangChain** → 第 5 层（API 层）。它是连接器，负责把其他层组织起来。
> - **向量数据库** → 第 2 层（架构层）。它是数据存储与检索的基础设施。
> - **Dify** → 跨越第 5 层和第 6 层。它既有 Workflow 编排（API 层），又有前端界面（应用层）。有时候 Dify 甚至可以替代 LangChain（都在第 5 层做流程编排），但 Dify 更强大的地方在它也包装了第 6 层。
> - 关系：LangChain 和向量数据库 → Dify 可以直接调用它们（比如在 Dify 里集成 Milvus，直接配置 LangChain 插件）。当你用 Dify 时，你其实隐含地在用 LangChain + 向量数据库 + 前端。Dify 是它们的"黑盒包装"。

## MCP 与 Function Calling 的层级关系

业界常见误解："MCP 出来了，Function Calling 该淘汰了。"

从分层角度可以清楚看到：MCP 处于**协议层（第 3 层）**，Function Calling 处于**应用层（第 5 层）**。MCP 是对 Function Calling 能力的**延展和标准化**，其底层仍然依赖 Function Calling 机制。协议层不能替代应用层，正如 HTTP 不能替代 Java——前者是通信规范，后者是逻辑执行单元。

## 六种 Multi-Agent 编排模式

这六种模式是现成的设计模板，类比网络编程中的 epoll/select 模型，无需从零发明：

**1. Prompt Chaining（提示链）**
模型输出结果决定下一步流程走向。最基础的条件分支——到达网关后判断是否需要再调用下一个模型，不需要则退出。

**2. Routing（路由模式）**
根据输入类型，通过路由机制决策流向哪个模型。典型应用：**意图识别**。案例：某快递公司用微调的小模型做意图识别，将用户请求路由至售前、售后、客户分析或转人工四个方向，替代原来单一大模型的处理，大幅降低时延。Dify、Coze 等工作流平台均内置此模式。

**3. Parallelization（并行模式）**
并发调用多个模型，汇总结论。工程实例：**多路检索 RAG**——同时用向量相似度检索和关键词（BM25）检索两条路，再对结果择优或合并，准确度高于单路检索。Dify、Coze、RAGFlow 等框架中的"多路检索"均是此模式的工程实现。

**4. Orchestrator-Workers（编排器模式）**
编排器（Orchestrator）负责任务分配，工作者（Worker）执行具体任务，最后汇总。即任务分拆。

**5. Evaluator-Optimizer（评估优化模式，ReAct）**
基于用户请求评估是否调用工具 → 调用 → 判断是否继续 → 形成思考闭环。这就是 ReAct 模式的本质。

**6. Autonomous Agent（自主代理模式）**
大模型与人持续交互，直到产出满意结果才停止。Manus、AI 编程自动运行等均属此类。此模式必须是 Agent 循环逻辑，前五种可以是 workflow（静态流）也可以是动态模型调用。

遇到 Multi-Agent 问题时，先套用以上六种模式之一，再用代码实现。

## 工程化典型挑战与解法

**挑战 1：让 AI 调用数据库**

| 方法 | 说明 | 问题 |
|------|------|------|
| 传统：Prompt 拼接 Schema | 直接把数据库结构写入 Prompt | 超出上下文窗口、内容写死、无法自动化 |
| Function Calling + SQL 生成 | 模型识别意图 → 生成查询 → 执行 → 返回结果 | 工程化起点 |
| RAG（向量化表结构） | 模糊搜索数据库表内容，语义召回 | 适合非结构化查询 |
| Tool/Agent 封装 | 数据库操作封装为工具或 Agent，调用时加权限控制和审计日志 | **推荐方案**，比 RAG 更易审计 |

**挑战 2：让 AI 分步解决复杂问题**

引入 TOT（Tree of Thought）或 GOT（Graph of Thought）进行任务分解 → 多 Agent 协作。任务编排（可视化流程）是方便人看的表达；代码层面对应**有向无环图（DAG）**，即 LangGraph 的子任务图实现。

**挑战 3：多个 AI 协同工作的组织方式**

以五角色 Multi-Agent 为例（Manager / Planner / Executor / Reviewer / Improver）：
- **流水线式**：A → B → C，固定顺序执行
- **并行式**：多个 Executor 同时执行，Review Agent 裁决，Human 可介入确认
- **辩论式（虚拟法庭）**：多个 Executor 提出方案，审核 Agent 评判，提出改进——适用于需要多方论证的决策场景

AutoGen（微软）、CrewAI、MetaGPT 等框架已内置上述模式，降低实现难度；但架构设计的复杂度相应提高。

## 传统 AI 开发 vs. AI 工程化开发

| 维度 | 传统 AI 开发 | AI 工程化开发 |
|------|-------------|--------------|
| 调用方式 | 调用一次大模型，堆在一起 | 在实效性保障前提下拆成多 Agent，各自负责专业职责 |
| Prompt 角色 | 硬编码输出风格（Hard-coded Prompt） | 流程控制——调起工具、调起模型、管理上下文交互 |
| 目标定位 | "能跑就行"的实验原型 | 可靠、可控、可观测、可携带的工业级系统 |
| 产出形式 | 单一模型服务 | 编排 + 监控 + 反馈的动态系统 |

工业级质量要求包括：稳定性、可维护性、可观测性、安全性、性能。其中**多 Agent 化**（在实效性得到保障的前提下）是 AI 工程化发展的核心趋势，预计至少在近一到两年内成立。

3. 对比传统 AI 开发和 AI 工程化开发，Prompt 的角色发生了什么根本转变？为什么这个转变能体现"工程化思维"？

<details>
<summary>答案指南</summary>

1. 八层架构的层级思维帮助工程师先定位问题所在层级再针对性解决：算力不足应在第 1 层（基础设施层）扩充硬件，而非在第 5 层（认知推理层）调参；负载均衡属于第 2 层（AI 智能体互联网层），而非业务逻辑层。层级错配会导致在错误位置空转。
2. Routing 模式通过意图识别将不同类型请求路由至对应模型，典型案例是某快递公司用微调小模型做意图识别，将用户请求分流到售前、售后、客户分析或转人工四个方向，大幅降低时延。Parallelization 模式并发调用多路检索（向量相似度 + BM25 关键词），再对结果择优合并，准确度高于单路检索。
3. 传统开发中 Prompt 是"硬编码输出风格"的内容生成工具；AI 工程化开发中 Prompt 升级为流程控制指令——负责调起工具、调起模型、管理上下文交互。这一转变体现了关注点分离：Prompt 不再只管"说什么"，而是管"系统怎么运转"。
