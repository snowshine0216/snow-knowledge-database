## AI 的新定义：工具化

AI 的定位正在从"高高在上的技术"转变为"胳膊和腿的延长"——一种工具化的存在。传统学 AI 的路径是先理解大模型工作方式，然后让人去适应 AI；而现在的趋势是让 AI 直接融入工作流、成为生产力工具。

讲师将这一转变称为 **"AI 的上班方式"**：AI 像员工一样被雇佣，承担具体的工作任务。当前主流的上班方式有三种（外加一种逐渐被淘汰的旧模式）：

1. **工作流模式**（已淘汰）：用固定提示词逐步驱动大模型，体现不出大模型的语义理解优势，效果往往不如传统 RPA。
2. **RAG**：让 AI 查资料（知识库）。
3. **AI Agent（单智能体）**：让 AI 做分析决策。
4. **Agentic AI（多智能体）**：让多个 AI 互相协作完成复杂任务。

---

## 三种主流 AI 开发范式

业务 AI 化的第一步是判断业务能否拆解为上述三种范式之一。如果业务过于复杂无法直接套用，应先将业务切分成更小的功能单元，再分别匹配适合的范式。

## RAG：让 AI 查资料

### 适用场景

RAG（Retrieval-Augmented Generation，检索增强生成）解决大模型本身的五类局限：

1. **知识截止日期**：大模型预训练后知识不再更新，无法回答实时或内部信息。
2. **幻觉问题**：在医疗、金融等高风险场景，幻觉不可接受，需从知识库中召回可信内容。
3. **领域/私有知识缺失**：大模型没有企业内部资料，需要注入特定知识。
4. **可解释性与溯源**：RAG 可返回依据来源，使回答具备可解释性。
5. **成本替代微调**：微调成本高、复杂度大；RAG 以较低成本达到类似效果。

### 架构原理

RAG 的核心逻辑：

```
用户提问 → 在知识库中检索相关内容 → 将{用户问题 + 检索结果 + 系统提示词}组合 → 大模型生成答案
```

知识库的建立是离线过程：加载多种格式文档 → 统一切片 → 向量化存储。在线检索时，用向量相似度找到最匹配的知识块并注入到 prompt 中。

### 意图空间 vs 知识空间

实际工程中最大的挑战是**用户意图空间与知识库知识空间的错位**：

- 用户说"如何申请办公电脑" → 意图是"入职流程"
- 知识库存储的是"入职规范文档"

两者存在偏差，需要通过技术手段（问题改写、文档切片优化、关键词匹配、向量召回优化等）将用户意图映射到知识空间。方向是：让中间的覆盖区域尽量向用户意图空间靠拢，同时过滤掉不相关的知识匹配。

### NL2SQL：RAG 的典型变体

自然语言转数据库查询（NL2SQL）是 RAG 的一个重要应用模式：

```
用户自然语言 → 从知识库（DDL/Schema）中召回相关表结构 → 大模型生成 SQL → 执行 → 返回结果
```

缓存机制（Cache）可以复用已执行过的 SQL，提升效率。这一模式可推广至：自然语言 → JSON/XML/代码，只要满足"输入是自然语言，输出是固定格式"即可套用 RAG 范式。

## AI Agent：让 AI 做分析决策

### 什么时候使用 Agent

Agent 的核心是赋予大模型**大脑（思考/规划）+ 手脚（执行）**的能力。仅有思考而无执行，不需要 Agent——直接与大模型对话即可（如 DeepSeek 直接输出旅游攻略）。触发使用 Agent 的判断标准：

> **必须同时满足：能思考规划 + 能执行工具调用**

### Agent 的四种核心能力

1. **思考与规划**：将复杂任务拆解为子任务，动态决定下一步动作。
2. **感知**：解析用户输入（文字/图片/语音）、感知系统状态和文件变化。
3. **执行**：调用 API、查询数据库、发送消息、操作本地文件。
4. **记忆**：存储对话历史和过往经验，支持长期决策和上下文延续。

感知和记忆是"加分项"，思考 + 执行是"必选项"。

### 典型应用模式

**分发聚合模式**：同一任务分发给多个不同模型（千问、DeepSeek、Kimi），各自输出分析结果后由聚合器汇总，用于重要决策场景。

**AI IDE 的 Agent 实现**（以 Tree/Cursor/Codebuddy 为例）：
- 思考：将需求拆解为功能模块
- 感知：感知代码变化，解析报错信息
- 执行：安装依赖库、运行程序、调用搜索接口
- 记忆：在多轮对话中保持最初需求的上下文

## Agentic AI：多智能体协作

当任务需要多个专家角色同时存在、各自维护独立记忆和主张时，单 Agent 会因上下文空间有限而表现不佳（一个人不能同时精通 Java + Go + C++ + Python）。此时应使用多智能体（Agentic AI）：

- **单 Agent 适合**：一个领域专家角色（如论文分析、代码审查、健身教练）。
- **多 Agent 适合**：多角色协作、记忆不能混淆（如狼人杀游戏、法庭庭审模拟、团队写作流程）。

团队写作示例：编辑 Agent + 主笔 Agent + 校对 Agent + 网络搜索 Agent，各自独立运行，通过消息协作完成一篇文章。

## 工程化视角 vs 科研视角

| 维度 | 科研视角 | 工程化视角 |
|---|---|---|
| 数据 | 干净标注数据 | 原始、有噪声、含隐私限制 |
| 目标 | 单次最优效果（追求 SOTA） | 稳定性、成本、可维护性 |
| 部署 | 本地 GPU 跑通即可 | 本地/云端，高并发/低延迟 |
| 迭代 | 发表论文后结束 | 持续打补丁、收集用户反馈回流 |

工程化不是一次性项目，而是动态演进的系统。讲师建议：先从 RAG 入手（过程透明、可解释、白盒），再逐步学习 Agent，最后掌握多智能体框架。技巧要"因场景而异"，不要一股脑全用上——业务不需要的能力加进来只会让系统越来越重。
