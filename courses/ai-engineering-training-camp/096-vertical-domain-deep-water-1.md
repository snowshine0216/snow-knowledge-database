## 背景：垂直领域的核心挑战

法律、金融、医疗三个垂直领域对 AI 产品的要求区别于互联网产品：

- **稳定性优先于创造性**：这些领域不希望 AI "胡说八道"，关键是可靠的信息回答
- **核心技术底座**：
  1. 成熟的 RAG 检索能力
  2. 完善的数据流水线
  3. 数据治理体系
- 提示词往往设计得非常长，确保生成链路稳健

---

## 法律场景：死穴一 — 力度错配

**问题描述**：将民法典、刑法等法律文本按固定 token 大小（如 512K）暴力切片后放入向量数据库，导致逻辑单元被破坏。

**具体表现**：
- 询问"防卫过当应该怎么判"，系统只返回刑法第2条（正常防卫）和第234条（故意伤害）
- 逻辑断裂：用户拿不到"情节较轻可免除处罚"这条关键但位置分散的条款
- 典型案例：合同无效需要综合民法典第144条、第146条、第555条三条来判断，单独向量检索极易漏掉

**根本原因**：法条之间存在严格的逻辑推导关系，A→B→C，单独提取 C 是无效的，而暴力切片破坏了这种单元性。

---

## 法律场景：死穴二 — 关键字丢失

**问题描述**：法律检索中大量使用精确标识符（如"2023清零一民众1234号"、"最高指导案例第100号"），这类含数字和编号的查询在向量数据库中极易出现错配或检索不到。

**根本原因**：向量/嵌入模型对数字和编号天然不敏感，纯语义检索无法处理精确的字符串匹配需求。

---

## 法律场景：工程解法

**解法一：结构化切分（解决力度错配）**

使用 LlamaIndex 的层级感知分割器，将法律文档转换为 XML / Markdown 层级格式：

```
条 → 款 → 项 → 目
```

- 用正则规则做预解析，识别"第几条第几款"
- 按语义边界（而非固定 token）切分，保证逻辑单元完整
- 在 Metadata 中存储层级关系和上下文，供律师追溯

**解法二：混合检索（解决关键字丢失）**

| 检索层 | 工具 | 作用 |
|--------|------|------|
| 关键词检索 | Elasticsearch / BM25 | 精确匹配编号、数字 |
| 向量检索 | MILVUS 等 | 语义相似度匹配 |
| 排序 | Re-rank 模型 | 融合两路结果 |

现在已有封装好的双路检索库，当时是手动集成 Elasticsearch 做关键词过滤，再用 re-rank 融合结果。

**提示词约束示例**：

```
基于提供的法律条文回答，格式必须为：
结论 → 依据（法第X条第X款，原文是...） → 参考文献
```

## 法律场景：进阶方案——知识图谱

当时未实现，但现在可用更优方案——**知识图谱 + GraphRAG**：

1. 从判决书中抽取三元组，例如：
   - 法律依据：刑法第20条第2款
   - 情形排除：正当防卫
   - 结果：伤害免责
2. 将三元组导入 Neo4j 构建知识图谱
3. 查询时用 NER 识别实体（如"打架"、"正当防卫"），做子图搜索
4. 将图谱召回的刑法条文与 RAG 召回内容合并成提示词

**效果**：用户问"打架还手算正当防卫吗"，可以准确返回"需满足三个条件：对方先动手、未超出必要限度、废除户欧"。

可用工具：
- DBKE（开源）
- LlamaIndex GraphRAG 模块
- LLM 提取三元组 → Neo4j 存储

## 法律场景：元数据锚定与版本控制

**元数据锚定**：每个切片的 Metadata 必须包含：
- 法律名称（如：民法典）
- 条款编号（如：第144条）
- 来源数据库标识

**版本控制**：
- 法律文书有生效时间和废止时间
- 检索时自动过滤已废止条文（在 Top-K 结果进入模型前，通过 filter 手动过滤）
- 法律更新时，直接向向量数据库添加新条目并设置生效/废止时间，无需重建索引

## 金融场景：非结构化数据处理

**背景**：金融行业 100% 的数据是非结构化数据（研报、合同、通话录音、社交媒体舆情），传统工具难以挖掘。

**数据问题的三个混乱来源**：
1. **格式混乱**：PDF、Word、图片混合
2. **内容混乱**：表格、图表、公式并存，跨页表格容易"撕裂"
3. **语义模糊**：专业术语（PE 估值、ROE、非标资产、GDP/CPI/PPI 等缩写）含义歧义

**文档解析方案（当时）**：
- 使用百度 PaddleOCR 做版面分析（分离标题/段落/表格/图表）
- PyPDF2 加载 PDF
- 表格重建：提取表格位置与内容（借助视觉方向博士）
- 图表理解：当时未成功，取巧方案——从图表对应的原始表格数据做分析，"假装"是对图表的理解

**语义标准化**：
- 整理金融常见术语 + 定义 + 关系，构建本地金融术语库
- 使用 BERT 模型做实体链接（entity linking），将缩写/歧义词映射到标准定义
- 当时用 one-shot prompt 方式，现在可通过 MCP 工具实现术语查询器（遇到专业术语时调用 MCP 获取解释）

## 金融场景：数值幻觉与沙箱执行

**问题**：直接让大模型计算数值（如财务指标），容易出现幻觉，计算结果不可信。

**解法：沙箱执行**

```
用户提问 → 意图识别（是否涉及计算）→ 参数提取 → 调用 Python 沙箱工具
→ 在沙箱中用 numpy/pandas 执行计算 → 返回精确结果 → 合并进模型上下文
```

- 工具：numpy + pandas 满足大部分金融计算需求
- 现代方案：无需单独做意图识别，直接给模型配置计算工具，模型自动调用；用 Pydantic 做参数验证

**客户反馈**：客户认为沙箱精准计算功能是最大亮点，优先于 AI 决策功能。

## 金融场景：进阶方案

**多元数据融合（Flink 实时计算）**：
- 将研报中的营收增长数据 + 舆情中的负面新闻 + 交易中的成交量变化做关联分析

**知识推理**：
- 例如：公司营收增长 15%，但上游原材料涨价 20%，推断利润下滑
- 需要构建公司投资关系知识图谱来支持此类推理

**金融 AI 定位**：
- 不是替代人，而是解放人
- 核心是：数据解析器 + 代码执行器 + 知识推理机的组合
- 投资决策最终仍需人工判断（Human in the loop）

**注意**：AI 炒股本质上是概率性事件，不能以历史数据验证其准确性——因为模型输出依赖输入数据，数据变化则结果变化。
