## The Integration Problem MCP Solves

Before MCP, every development team that wanted to build an application with access to external data or services had to write their own integration code from scratch. A team building a product that needed to read from Slack, Google Drive, GitHub, and a Postgres database would write four separate wrappers — one per service — tailored to their application's needs. When a different team built a different product that also needed Slack and Google Drive, they repeated the same work independently. There was no shared standard, no reusable layer, and no way to leverage what anyone else had already built.

This duplication was not merely inconvenient. It meant that the total amount of integration work borne by the developer community scaled as the product of the number of applications and the number of data sources: if M teams each needed to integrate with N tools, the community as a whole performed M×N integrations.

## From M×N to M+N

MCP, the Model Context Protocol, was proposed by Anthropic as a standard way for applications to gain access to tools and data sources. By agreeing on a common protocol, both the application side and the data-source side only need to implement the standard once. Each of the M applications implements the MCP client interface once, and each of the N tools or data sources implements the MCP server interface once. The total work drops from M×N to M+N.

The initial focus of MCP was on context enrichment — giving an LLM access to more information by fetching data from external systems. The MCP documentation calls these read-only data access primitives "resources." Over time, the protocol expanded beyond passive data retrieval to cover the full range of actions an application might want to take, making MCP a general-purpose integration layer rather than a pure data-fetching mechanism.

## Clients and Servers in the MCP Ecosystem

MCP defines two roles. An MCP client is the application that wants to use tools or access data — it consumes capabilities exposed by the other side. An MCP server is the software that wraps a data source or external service and exposes those capabilities to any compliant client. A GitHub MCP server, for example, wraps the GitHub APIs and exposes operations like reading files from repositories or listing pull requests. A Slack MCP server wraps the Slack APIs and exposes messaging and channel data.

Because the protocol is open, a growing number of both clients and servers now exist across the ecosystem. Any application that adopts MCP automatically gains access to the full catalogue of available MCP servers, and any new MCP server immediately becomes available to every existing MCP client. The network effect compounds as the ecosystem grows.

## MCP in Action: A Live Demo

The lecture demonstrates MCP using the Claude Desktop application connected to a GitHub MCP server. When the user asks "summarize the README from the GitHub repo at this URL," the application — acting as an MCP client — sends a structured request to the GitHub MCP server asking it to retrieve the file. The server fetches the content and returns it; that content is fed into the LLM's context window, and the model produces a clean summary.

A second query, "what are the latest pull requests," causes the LLM to invoke a different tool on the same server: a list-pull-requests operation with parameters specifying the repository, sort order, and result count. The server returns the raw data, and the model writes a readable summary of recent activity. The entire flow — tool selection, parameter construction, server call, response incorporation — happens transparently within a single conversational turn.

## Why MCP Matters

MCP represents a structural shift in how the AI application ecosystem handles integrations. Rather than each team solving the same connection problems in isolation, MCP creates a shared vocabulary and a shared marketplace of capabilities. Developers who learn the protocol gain immediate access to a large and rapidly expanding library of MCP servers built by others, and developers who publish MCP servers make their integrations available to every MCP-compatible application at once.
