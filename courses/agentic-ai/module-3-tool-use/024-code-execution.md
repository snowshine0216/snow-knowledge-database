## Motivation: Beyond One Tool Per Operation

A common early approach to building LLM-powered tools is to expose one function per operation — an `add` tool, a `subtract` tool, a `multiply` tool, and so on. This works for simple, predictable requests, but it breaks down immediately when a user asks for something not covered by the existing tool set. Asking for the square root of two, for instance, would require writing yet another tool. A modern scientific calculator has dozens of operations, and the long tail of mathematical tasks a user might request is effectively unbounded. Creating a separate tool for every capability is an engineering dead end.

A far more powerful alternative is to give the LLM the ability to write and execute code. Rather than anticipating every operation in advance, the agent generates a small program on the fly, runs it, and returns the result. This single capability subsumes an unlimited number of specific tools.

## How Code Execution Works

The mechanism is straightforward. A prompt instructs the model to express its solution as executable Python, wrapped in a pair of delimiters — for example, `<execute_python>` and `</execute_python>` tags. When the model responds, the application uses pattern matching (typically a regular expression) to locate those tags and extract the code block between them.

For instance, given the query "What is the square root of two?", the model might emit two lines of Python that import the `math` module and call `math.sqrt(2)`. The application captures those lines, executes them, and obtains the numerical result `1.4142...`. That result is then fed back to the model, which formats it into a human-readable answer for the user. The whole loop — prompt, generate, extract, execute, respond — is transparent to the end user.

## Execution Mechanisms

There are two main ways to run the code the model produces.

The first is Python's built-in `exec` function. It accepts a string of arbitrary Python code and executes it in the current process. This is simple, powerful, and requires no external dependencies. The obvious downside is that the code runs with the same permissions as the host application, which creates real security exposure.

The second approach is to use a sandbox environment. Tools such as Docker containers or lightweight services like E2B provide an isolated execution context. Code runs inside the sandbox and cannot reach the host filesystem, environment variables, or network resources unless explicitly permitted. This significantly reduces the blast radius if the model generates dangerous or incorrect code.

## Recovering from Code Errors

LLMs do not always produce syntactically or logically correct code on the first attempt. A robust agentic application treats code execution failure as a recoverable signal rather than a terminal error. When `exec` (or the sandbox) raises an exception, the error message and traceback are passed back to the model along with a prompt asking it to reflect on what went wrong and try again. Allowing one or two retry iterations in this way — a pattern related to the reflection techniques covered earlier in the course — is often enough to arrive at a correct solution that would have otherwise been missed.

## Security Risks and Sandbox Environments

Running arbitrary LLM-generated code carries real, non-hypothetical risk. A concrete example: a highly agentic coding assistant once executed `rm -rf *.py` inside a project directory, deleting multiple Python source files. The model later acknowledged the action was a serious mistake, but the files were gone. In that instance the developer had a GitHub backup and no permanent harm resulted, but the incident illustrates how quickly things can go wrong when generated code executes with full host permissions.

Best practice is to run LLM-generated code inside a sandbox. In practice, many developers use `exec` directly without sandboxing, accepting the risk for speed and simplicity. For production systems or any environment where data loss or sensitive-data leakage would be harmful, a sandbox — Docker being the most common choice, E2B as a lighter-weight alternative — is strongly recommended. The sandbox does not eliminate the risk of bad code being generated; it limits the consequences if it is.

LLM providers are aware of how critical this capability is and invest significant effort in training their models to write safe, correct code. That training does not replace the need for infrastructure-level sandboxing, but it does raise the baseline quality of generated programs.

## Bridge to MCP

Code execution is one of the most powerful tools an agent can have, and adding it to an application dramatically expands what the application can do. The next step beyond managing individual tools is a standard way for developers to share and discover tool sets without reimplementing them from scratch. That standard is the Model Context Protocol (MCP), a recently introduced specification that is rapidly gaining adoption across teams building LLM-based applications. The next lesson introduces MCP and explains how it fits into the broader agentic tool-use picture.
