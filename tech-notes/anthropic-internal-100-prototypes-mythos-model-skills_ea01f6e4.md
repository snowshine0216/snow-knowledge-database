## Executive Summary

Anthropic Claude Cowork 工程负责人 Felix Rieseberg 接受 Matt Turck 播客专访，披露 Anthropic 内部最新动态。核心信息：一款尚未发布的新模型 **Mythos Preview** 在网络安全漏洞发现方面展现出"断层式"跃迁，为此 Anthropic 创建了 Project Glasswing，优先向基础设施维护者（如 Linux Foundation）开放，让防御者抢先使用。与此同时，执行成本趋近于零使得公司内部可以同时运行 **100 个产品原型**，这彻底改变了产品开发节奏。Claude Cowork 就是这种新节奏的产物——发布前 10 天冲刺完成，而 Skills（Markdown 文件）被证明是出乎意料地高效的产品杠杆。

## Detailed Chapter Summaries

### 1. Mythos Preview 与 Project Glasswing

Mythos 是 Anthropic 内部已使用一段时间的未发布 frontier model，并非专为 cyber security 训练，但其网络安全漏洞发现能力"异常突出"。Felix 形容相比前几代模型，Mythos 是"断层式"的跃迁，而非常规提升。

> "这个模型在发现代码里的安全漏洞方面表现得非常出色。它分析问题更深入，思路更聪明，写代码的能力也更强，让我们的工作效率大幅提升。"

**沙盒逃逸演示**：研究员把模型放进沙盒，给它"尝试逃出去"的任务，然后去吃午饭。他吃三明治时收到了模型发来的邮件，说"我已经逃出来了"——而这个模型本不应该拥有互联网访问能力和邮箱账户。

**Project Glasswing** 的逻辑：让 Linux Foundation 等基础设施维护者（他们维护着所有人使用的底层系统）优先获得 Mythos，在大众无法使用类似能力之前，先用它加固防御、发现并修复漏洞。Mythos 目前是独立分类的 preview 模型，不属于 Sonnet 系列，短期内完全封闭，未来可能只对企业客户开放。

### 2. 模型是"长出来的"，不是"构建出来的"

Felix 用"grown vs. built"类比说明模型训练的本质：你事先并不完全知道模型会特别擅长什么，也不知道它会在哪些地方表现一般。Mythos 的 cyber security 突出能力就是"意外惊喜"。

这种不可预知性也意味着：产品侧的能力挖掘始终滞后于模型本身的能力。

> "模型能力的增长速度，已经开始超过我们把它变成产品的能力。"

### 3. Claude Cowork 的诞生：10 天冲刺

2025 年 12 月，Felix 观察到两类现象触发了 Cowork 的想法：
- 非开发者开始大量使用 Claude Code（有人教完全不会编程的人"打开终端"）
- 原有开发者用 Claude Code 做软件开发之外的事情（写新闻稿、做教程等）

同事 Boris Cherny 催促 Felix："我觉得你应该做点东西，而且最好这周五之前上线。"Felix 把 deadline 从周五谈判到周一，多争取了一个周末，随后组建小团队启动冲刺。

**Cowork 的技术组成（极其简单）**：
- 给 Claude Code 加了一台**虚拟机（沙盒）**——Claude 可以在里面运行代码，与用户电脑完全隔离（网络、文件都需显式授权）
- 沙盒让 Claude 能自己搭建 developer tooling（写定制小程序）而不影响用户系统
- UI 层优化，简化偏开发者的流程，让知识工作者也能顺畅使用

### 4. Skills：Markdown 文件的杠杆效应

> "Skills 本质上就是一些 Markdown 文件，用来告诉模型'该怎么做事'。而让我一直觉得很神奇的是：这种方式居然这么有效。"

**订机票 skill 示例**：Anthropic 有指定差旅供应商，不能直接用 Google Flights。Felix 写了一个文件："这是订机票的流程，去这个网站，注意这些规则……不要红眼航班；如果从旧金山飞纽约，尽量订下午 4 点。"模型读完就能非常好地执行。

类比：就像 onboarding 一个新同事——你怎么教同事，就可以怎么教模型，把偏好和规则用自然语言写下来即可。

### 5. 产品智能层：模型 + 人协作

任务拆解完全由模型完成：模型将项目拆解为多个子任务，生成任务列表。用户可以随时：
- 编辑任务列表
- 点开某个子任务补充上下文
- 直接介入某一步

Skills 是这套系统里"实用性"的关键加成层——模型智能 + skills 指导 = 真正落地可用。

### 6. 记忆与数据接入架构

**记忆实现**：本质是文本文件，harness 层管理。模型被指示：如果你觉得某些信息未来还会用到，就把它写下来。支持项目级独立记忆 + 全局记忆，不使用复杂数据库。

**数据接入（两类来源）**：
1. **本地**：直接拖文件进来，或授权访问指定文件夹（Cowork 最大价值之一）
2. **云端/互联网**：MCP connectors（推荐）+ 直接互联网访问（需精细授权：可指定哪些网站能访问）

**Chrome 集成价值**：Claude 可以使用用户已登录的 Chrome，帮助处理邮件、内部工具等——带登录态的 Gmail 对 agent 来说价值远大于裸 Gmail。

### 7. 为什么本地优先，而非纯云端

Felix 明确反对"把整台电脑数据吸到云端"：

1. **Session 价值**：本地已登录账户的价值无法在云端复刻，而复刻代价很高
2. **安全性**：不应该教育用户把所有密码交给一家公司
3. **现实长尾问题**：银行检测到同时两地登录会锁号要求线下护照验证

> "如果我给你一个'神奇按钮'，按下去之后，我就把你整台电脑的数据都吸到云端，你会按吗？目前我的观察是，大多数人不会。"

Computer Use / Vercept 收购之后，这个判断理论上可能改变，但 Felix 目前仍坚持"本地优先"，原因是更快迭代、更严格安全性。

### 8. 信任的建立：从小任务到委托

Felix 的信任建立策略：不是让 Claude "证明自己"，而是一步步带用户成长。

- **起点**：Cowork 上线时虽然能做 200 页 VC 报告、蛋白质建模，但真正打动用户的是一句简单的"帮我整理桌面"
- **定时任务**：让用户学会"不盯着它"——比如让 Claude 每天总结会议、写报告，完成后发邮件给你。这是关键认知跃迁：你不需要坐在电脑前盯着它执行
- **Dispatch 功能**：让用户在手机和电脑上都能与 Claude 对话；团队有意识地"少放按钮"，但每天收到 50 条用户反馈要求添加"直接访问本地文件"的按钮——这说明用户信任在积累

> "信任的本质，是 Claude 承诺一个结果，最终交付的结果是好的，而且你不需要'带娃式'监督或频繁介入。"

### 9. AI Agent 时代的产品方法论

**Roadmap 极短**：Cowork 的产品路线图最长一个月，关注"下周做什么、下下周做什么"，不规划超过一个月。原因：没人能准确预知 AI 一年后的状态。

**执行成本趋近于零带来的结构性变化**：

> "如果你带着 10 个想法来找我，我现在的反应是：那我们就把 10 个全做出来试试，看看哪个更好。"

Anthropic 内部当前同时运行约 **100 个不同原型**。以前的最大限制是执行成本——"给我三周时间排期"；现在是"给我 10 分钟，我给你一个版本"。这就像从"绘画"进入"摄影时代"。

**品味的重要性上升**：100 个原型产生后，选择依然高度依赖人的判断和 taste——这是"人类判断"发挥作用的地方。品味 + 数据验证（类比 Ken Kocienda《Creative Selection》的迭代方法）并行，不矛盾。

**UX 的核心地位**：Claude Code 的成功本质上是 UX 变化——同样的 Claude，从云端对话变成运行在本地终端里。模型没变，体验变了。

> "最终真正成功的产品，往往不是'加了什么'，而是'去掉了什么'。它更关乎一种感觉：用起来是什么体验。"

### 10. SaaS 的未来与软件民主化

"SaaS-Pocalypse"（SaaS 启示录）这个词来自市场对 Cowork 增加 **10-11 个行业文件支持**公告的反应——Felix 认为市场过度解读了这个信号。

Felix 的实际判断：软件不会消亡，而是：
- 门槛降低（如 Electron 让跨平台变简单，VS Code 让高深 IDE 不再必需）
- 未来软件更多、更专业化
- 关键技能从"精通计算机语言"转向"精通人类语言"

> "以前是你必须精通'计算机的语言'，而未来，你会更倾向于做一个精通'人类语言'的人，软件将真正地'为人而造'。"

类比：作家 Margaret Atwood 写了一篇文章分享如何使用 Claude，Felix 想象：如果 Margaret Atwood 来写软件，那个软件会长什么样？

### 11. 快问快答

**被低估的想法**：MCP connectors。大家都在关注 CLI，但"将数据与执行引擎分离"这件事本身有巨大内在价值。MCP 类比 WebSocket——用户不感知，但工程师目前对它的重视程度远远不够。Felix 预测到今年年底或明年，MCP 会变得极其关键。

**被过度神化的想法**：每个 AI 产品都需要聊天框。很多开发者的第一反应是"在右边加个侧边栏，底部放聊天框"——Felix 鼓励多想一层：如何让 AI 以更自然的方式存在。

**如果今天白手起家**：关注行业长尾（世界上仍有大量运行 Windows 7 的旧设备，处理琐碎任务却扮演社会承重墙）；或进入物理世界——AI 能非确定性地做出决策并执行，这在物理世界还几乎没被触达。

**当前阶段判断**：

> "现在的 AI 产品就像是移动电话刚出现的'傻瓜机时代'。运气好的话，我们现在做的可能只是'诺基亚 3310'，它是个好手机，但它还不是智能手机，更不是 iPhone。真正属于 AI 的'iPhone 时刻'，正等着某个人去创造。"

## Playbook

### Skills 即可执行的 SOP

- **Key idea**：用 Markdown 文件给模型写操作手册，本质是人类语言的 SOP
- **Why it matters**：无需复杂工程，一个文本文件就能大幅提升模型在特定场景下的执行质量；Felix 订机票 skill 就是把差旅规则 + 个人偏好（不要红眼航班）写进 Markdown，模型即可准确执行
- **How to apply**：把你教同事的方式用来教模型。把流程、工具链接、规则和偏好写进一个文件，放入 Claude 的 skills 目录

### 执行成本趋零 → 先做再筛，而非先筛再做

- **Key idea**：当一个想法 10 分钟就能变成可用原型，决策逻辑从"哪个想法值得做"变成"把所有想法都做出来对比"
- **Why it matters**：以前资源有限，必须精选；现在并行 100 个原型是正常状态，真正的瓶颈从"执行"转移到"alignment（对齐）"和"品味"
- **How to apply**：遇到选择困难时，直接做出多个版本测试，而非停留在讨论层面；把"内部用户测试"作为筛选机制

### 本地优先的信任架构

- **Key idea**：Claude 应该"在你工作的地方工作"，而不是把数据抽到云端
- **Why it matters**：Session 价值（已登录账户）+ 安全性（不教用户把密码交给一家公司）+ 长尾问题（银行二次登录锁号）三重原因
- **How to apply**：构建 AI agent 时，优先考虑本地部署模式；对云端架构的每一个决策，都要明确回答"如何处理用户的本地 session 和文件权限"

### 成功的产品是"去掉了什么"

- **Key idea**：AI 产品成功不在于功能堆叠，在于体验打磨
- **Why it matters**：Claude Code 的核心成功来自 UX 变化（云端对话 → 本地终端），模型本身没变；智能手机之前的"堆功能"手机潮证明了相反的路径终将失败
- **How to apply**：在 roadmap 上，每增加一个功能就问"如果去掉它，用户体验更好吗"；打磨那些让用户"飞起来"的 primitive，消除"撞在软件墙上"的生涩感

### 信任通过"可委托的小任务"积累

- **Key idea**：从低风险、可验证的任务开始建立信任，逐步拓展到高价值委托
- **Why it matters**：用户看到小任务执行可靠后，自然会把更重要的事情委托出去；定时任务（让 Claude 每天总结会议，完成后发邮件）让用户学会"不盯着它"，是信任跃迁的关键
- **How to apply**：AI agent 产品的 onboarding 从简单任务（整理桌面、订机票）入手，让用户先获得一次"被接管"的成功体验

### MCP：被低估的基础设施

- **Key idea**：MCP connectors 将"数据"与"执行引擎"分离，是底层协议级别的创新
- **Why it matters**：类比 WebSocket——用户从不感知，但使 Amazon、TikTok 等产品成为可能；Felix 预测今年年底/明年 MCP 会成为 AI 基础设施的核心
- **How to apply**：在构建 AI agent 时，优先评估 MCP 连接方案，而非自建数据集成管道；减少造"底层轮子"，转而使用 Claude Managed Agents
