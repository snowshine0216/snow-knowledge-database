# NL2SQL Deep Dive Part 1

深入讲解 NL2SQL 的核心挑战、适用场景、语言歧义类型，以及基于 RAG 思路的开源框架 Vanna 的使用与架构设计。本讲是 AI 工程化训练营第 59 讲，对应模块 6 后半部分。

## Key Concepts

- **NL2SQL vs Text2SQL**：NL2SQL 是泛指所有自然语言转 SQL 的概念（含语音、手写），Text2SQL 专指文本对话形式，范围更窄。本课中两词等同。
- **行业准确率上限**：目前最先进方案在 Spider 数据集上准确率约 60%~70%，无人敢承诺超 80%，多表联合查询准确率低于 70%。
- **适用场景**：简单 SELECT/WHERE 查询和聚合查询适合 AI 生成；多表联合查询（JOIN）可靠性太低，不建议用于生产。
- **三类语言歧义**：词汇歧义（同词多义）、句法歧义（结构解读不唯一）、语义欠指定（信息不完整）。处理策略均为追问用户。
- **发展四阶段**：规则 → 注意力机制 → 预训练微调（50%）→ 当代大模型（60~70%）。
- **Vanna 框架**：基于 RAG 的 NL2SQL 开源框架，核心是把表结构、历史 SQL、文档存入向量数据库，查询时检索增强提示词再生成 SQL。支持多继承方式灵活替换大模型和向量 DB。
- **语法校验兜底**：所有 Text2SQL 工具生成 SQL 后必须先做语法校验，不通过则打回大模型重新生成。
- **人工确认闭环**：用户确认 SQL 结果正确后自动入库向量数据库，使相似问题的后续回答更准确。
- **AI 取数场景**：最适合 NL2SQL 落地的是"AI 取数"——让 AI 生成用于日常报表/周报/月报的常规 SQL，而非处理陌生复杂查询。

## Key Takeaways

- 不要挑战多表联合查询，优先用其他手段实现，最后才考虑 Text2SQL
- Vanna 设计模式值得学习：train + ask 两个接口封装完整 RAG 逻辑；文档中直接提供各种模型/数据库组合代码
- 用户确认的正确 SQL 应回流向量数据库，形成数据飞轮
- Text2SQL 的两大路线：RAG（白盒、可控、目前主流）和微调（效果提升有限，成本高）
- 发展史阅读价值：通过论文了解哪些技术路线已被证伪，避免重踩
