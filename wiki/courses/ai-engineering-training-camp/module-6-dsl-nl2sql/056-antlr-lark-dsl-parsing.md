# Parsing DSL Grammar with ANTLR and Lark

DSL (Domain-Specific Language) parsing is a key technique for building AI agent systems where LLMs generate or modify structured configuration files that drive downstream tools, devices, or workflows. This note covers the two primary Python-accessible parser ecosystems — [[lark-parser]] (Lark) and [[antlr4]] (ANTLR4) — and how to wire them into a [[langgraph]] pipeline for safe, LLM-driven DSL mutation.

## Core Idea

In an agent system, mutable configuration lives in a DSL file. Core logic (e.g., a compiled C driver, a Python service) is immutable. The LLM only edits the DSL. A grammar validator is the last line of defense against hallucinated or malformed output.

## DSL Design Principles

- **Declarative**: describes *what*, not *how*. Implementation details stay in code.
- **Readable**: domain experts (non-engineers) must be able to verify it.
- **Scoped**: only fields that legitimately change are exposed in the DSL.
- **Versioned**: each mutation bumps a version field; the old version is backed up.

## Lark (Python, lightweight)

- Library: `lark-parser` (≥ 1.1.7), pure Python.
- Grammar defined in a `.lark` file (generated by LLM from DSL sample).
- `Lark(grammar).parse(text)` produces a parse tree.
- A `Transformer` subclass converts parse-tree nodes to Python dicts / JSON.
- Error messages pinpoint exact line and column — essential for semi-technical editors.
- Best for: Python services, development tooling, grammar iteration.

## ANTLR4 (multi-language, production)

- Grammar defined in a `.g4` file; `antlr4` CLI generates lexer/parser source code.
- Supports Python, Java, Go, C#, JavaScript, and more.
- Compiled performance; exception-based error reporting with line info.
- Best for: industrial systems, multi-language pipelines, performance-critical validation.
- Requires JVM toolchain for grammar compilation.

## LangGraph Integration (7-step pattern)

1. Define shared `State` (user input, current DSL, updated DSL, validation result, reply).
2. **Intent recognition** — keyword match or LLM call to identify which field(s) to change.
3. **DSL update** — LLM call with strict prompt: modify only the target value, preserve syntax, return raw DSL, no markdown fences.
4. **Grammar validation** — run Lark parser on the updated DSL; fail fast with error detail.
5. **Apply changes** — write validated DSL to disk, back up previous version.
6. **User input handler** — route normal changes through pipeline; route out-of-range requests to confirmation branch.
7. **Build StateGraph** — add nodes/edges, compile, invoke.

## LLM Accuracy Notes

- Single-field DSL changes: high accuracy across models (Qwen Turbo, GPT-4, etc.).
- Multi-table SQL generation: GPT-4 ~60%, fine-tuned models ~80% — too low for safety-critical use.
- Grammar validation compensates for LLM hallucination; without it, corrupted DSL can reach production.
